SYNAPSE GUILD — REHYDRATION PACK (CHECKLIST)
Version: v1.5
Last Updated: 2026-03-13
Status: Reference (governance-aligned validation checklist; non-executable)

================================================================================
PURPOSE
================================================================================

A **Rehydration Pack** is the minimum complete bundle of subject-state artifacts required to resume
work in a new session (human or AI) WITHOUT drift, re-litigation, or loss of context.

This checklist is binary:
- If every required item is present and readable: REHYDRATION PACK = COMPLETE
- If any required item is missing, duplicated in an active slot, ambiguous, or unprovable:
  REHYDRATION PACK = INCOMPLETE
  → Disclosure Gate MUST trigger and execution authority is INVALID.

Authority note (anti-confusion):
- Governance pack files define law and templates.
- The Rehydration Pack contains the ACTIVE subject-state instances (in <Subject>_Data/...).
- Conversation is not an artifact. If the file does not exist on disk, it does not exist.

================================================================================
SCOPE
================================================================================

This checklist applies to continuing subjects (i.e., <Subject>_Data exists).

It does NOT replace:
- Processes/SYNAPSE_GUILD__CONTROL_SYNC_CHECKLIST.txt

Relationship:
- This checklist defines what must exist for rehydration to be real.
- The Control Sync Checklist is how you ignite the session after rehydration is proven.

================================================================================
CANONICAL LOCATION
================================================================================

Active pack location:
- <Subject>_Data/Latest Rehydration Pack/

Canonical archive location:
- <Subject>_Data/Archive/Latest Rehydration Pack/

Rule:
- Rehydration Packs are subject-state artifacts. They do not live in Governance.

================================================================================
ACTIVE-SET RESOLUTION RULE
================================================================================

“Latest” is not determined by guessing.
The active Rehydration Pack is resolved by explicit Subject-state pointers first.

Deterministic order:
1) Subject-state pointer(s) (for example in SUBJECT_STATE.yaml)
2) The single active artifact(s) present in the active pack location
3) Naming/date tokens only as a storage aid when no ambiguity exists

If multiple active candidates exist for the same slot and no pointer can lawfully resolve them:
- STOP
- trigger Disclosure Gate
- repair the active set before proceeding

================================================================================
REQUIRED CONTENTS
================================================================================

If an item is not applicable, it MUST be explicitly marked as NONE (not silently omitted).

1) Latest Rehydration Pack (active continuity)
Location:
- <Subject>_Data/Latest Rehydration Pack/

Required:
[ ] Exactly one active Bootstrap Prompt exists
    - MUST exist on disk.
    - MUST be unambiguously identifiable as a Bootstrap Prompt.
    - Subject-state pointer (if present) MUST resolve to it.

[ ] Exactly one active Continuity Lock exists
    - MUST exist on disk.
    - MUST be unambiguously identifiable as a Continuity Lock.
    - Subject-state pointer (if present) MUST resolve to it.

[ ] If an active Execution Pack exists:
    - exactly one active Execution Pack pointer exists under:
      <Subject>_Data/Latest Rehydration Pack/Execution Pack/
    - the pointer resolves to a canonical active Execution Pack source under:
      <Subject>_Data/Docs/Execution Packs/Active/

[ ] If no active Execution Pack is required:
    - no active Execution Pack pointer is required
    - the pack MUST NOT fabricate one

2) Buffs (session ignition key)
Location:
- <Subject>_Data/Buffs/

Required (see LOCK__SUBJECT_DATA_SKELETON.md):
[ ] <SUBJECT>_EXECUTION_PROTOCOL.txt
[ ] <SUBJECT>_DATA_DIRECTORY_MAP.txt
[ ] <SUBJECT>_SESSION_START_CHECK.txt

Rule:
- If Buffs are missing, the session MUST NOT “wing it.” Stop and repair Buffs.

3) Snapshots (ground-truth history)
Location:
- <Subject>_Data/Snapshots/
  - Control Sync/
  - End of Day/
  - General/

Required minimum:
[ ] Latest relevant Control Sync Snapshot(s) exist on disk when formal binding decisions were made.
[ ] Latest relevant End of Day Snapshot(s) exist on disk when prior work crossed a day boundary or sealed a day.

Rule:
- If the previous session performed work that should have been sealed and no snapshot exists, continuity is incomplete until repaired.

4) Codex / Canonical Vision (what execution must obey)
Location:
- <Subject>_Data/Codex/

Required:
[ ] Codex truth surface exists in a lawful state for the current Subject phase.
[ ] If CODEX_FREEZE.md exists:
    - Fog is lifted; governed execution may be legal.
    - Freeze is binding; execution must comply.
[ ] If CODEX_FREEZE.md does NOT exist:
    - Fog of War remains active.
    - Governed execution is forbidden.
    - Accepted Quests MUST be NONE.

Clarification:
- Drafting, discovery, and ambient structure formation may still exist while Fog is active.
- The Rehydration Pack must tell the next session which state is true now.

5) Active scope + commitments (only when applicable)
Guild Orders:
[ ] If a Raid is in progress:
    - <Subject>_Data/Guild Orders/ACTIVE/ contains the active Orders.
    - Otherwise: NONE.

Accepted Quests:
[ ] If any Accepted Quests exist:
    - <Subject>_Data/Quest Board/Accepted/ contains them.
    - Any required audit bundles exist under <Subject>_Data/Audits/Execution/.
    - Otherwise: NONE.

6) Engine / working state (the thing being built)
Required:
[ ] The current working state is accessible and corresponds to the Subject state.
    - Code subjects: repository present + updated.
    - Writing subjects: manuscript/assets present.
    - Mixed subjects: both present.

================================================================================
OPTIONAL / CONTEXTUAL CONTENTS (EXPLICIT)
================================================================================

Include only when required by the work state:

[ ] Execution Pack pointer (only when an active Execution Pack exists)
[ ] REHYDRATE.md or equivalent current-state orientation surface
[ ] Research notes / diagrams / temporary specs required to execute without re-litigation
[ ] External pointers required for execution (NEVER include secrets; store pointers only)

================================================================================
WHAT A REHYDRATION PACK IS NOT
================================================================================

A Rehydration Pack is NOT:
- a summary that replaces artifacts
- a substitute for <Subject>_Data/
- a substitute for Governance
- a license to invent new decisions during resume
- a pile of “recent files” guessed by timestamp alone

================================================================================
FINAL RULE (BINDING)
================================================================================

If resuming work causes confusion, drift, missing context, duplicate active artifacts, or re-litigation:
the Rehydration Pack is incomplete.

Correct response:
- STOP
- Trigger Disclosure Gate
- Repair missing/weak/duplicate artifacts (update Bootstrap Prompt / Continuity Lock / Snapshots / pointers as required)
- Re-run Processes/SYNAPSE_GUILD__CONTROL_SYNC_CHECKLIST.txt
