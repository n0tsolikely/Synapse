SYNAPSE GUILD — EXECUTION PACKS
Version: v1.5
Last Updated: 2026-03-13
Status: Governance Law (binding concept; pack contents are situational)

Maintainer: Brains (Peter) with Hands (Ash)
Audience:
- Human operators using Synapse Guild
- AI executors resuming bounded work
- Anyone handing off drift-sensitive execution across chats/operators

================================================================================
0) PURPOSE OF THIS DOCUMENT
================================================================================

This document defines what an Execution Pack is, when it is warranted, how it becomes active,
and how it participates in rehydration.

An Execution Pack exists for one purpose:

> When work is bounded, drift-sensitive, or handoff-heavy, the Execution Pack makes execution
> deterministic enough that another operator can continue without improvising the wrong thing.

Governance note:
- This document defines Execution Pack law and quality bar.
- Active Execution Packs are subject-state artifacts inside <Subject>_Data.
- The pack is execution law for a bounded window. It is not a new constitution.

================================================================================
1) DEFINITIONS (CANON)
================================================================================

1.1 Execution Pack — definition
An Execution Pack is a bounded execution artifact (single file or pack directory) that constrains
HOW a specific window of work is to be performed so that the operator cannot casually drift from
what was decided.

One-sentence definition:
An Execution Pack is a bounded, execution-facing pack that converts settled direction into
repeatable execution constraints.

1.2 Scope law
Execution Packs are local to a bounded execution window.
They do NOT redefine the whole Subject.

1.3 Authority
During its active window, an Execution Pack is binding for the work it governs.
If a pack conflicts with chat improvisation, the pack wins.
If a pack conflicts with higher law (Truth Gate, Disclosure Gate, Codex Freeze, etc.), higher law wins.

================================================================================
2) THE CORE INTENT (WHAT AN EXECUTION PACK DOES)
================================================================================

An Execution Pack exists to remove the wrong freedoms.

It answers questions like:
- What is the bounded objective?
- What is explicitly out of scope?
- What must not be changed?
- What must be verified before this window counts as done?
- What artifacts become the new truth afterward?

A good Execution Pack makes execution boring.

================================================================================
3) WHEN AN EXECUTION PACK IS WARRANTED (TRIGGERS)
================================================================================

An Execution Pack is warranted when any of the following are true:
- execution spans multiple chats/operators
- the work is drift-sensitive
- the work has strict boundaries or forbidden refactors
- verification must be carried along with the task
- the work is resumed after a delay
- a Control Sync explicitly says deterministic execution is required

An Execution Pack is NOT required for every movement in the system.
If the bounded window can be executed safely from current active truth alone, no pack is required.

================================================================================
4) WHAT AN EXECUTION PACK IS NOT
================================================================================

An Execution Pack is NOT:
- a Continuity Lock
- a Bootstrap Prompt
- a Quest Board entry
- a Snapshot
- a Codex section
- a Build Manual chapter

It does not replace any of those.
It constrains a bounded execution window.

================================================================================
5) SHAPE AND ALLOWED FORMS (NO TEMPLATE LAW)
================================================================================

Execution Packs are intentionally flexible.
They may be:
- one file
- multiple files in a pack directory
- a main file plus appendices

Law:
- flexibility of form is allowed
- ambiguity of authority is not allowed

Every valid pack must still satisfy MVEP (Section 6).

================================================================================
6) MINIMUM VIABLE EXECUTION PACK (MVEP)
================================================================================

Every Execution Pack must answer these questions.
If a pack does not answer them, it is incomplete.

6.1 Identity
- What is this pack called?
- What bounded scope does it govern?
- What triggered it?

6.2 Objective
- What must be true when this pack is complete?
- What is explicitly out of scope?

6.3 Inputs / prerequisites
- What artifacts/repos/zips must be loaded?
- What must be read first?

6.4 Execution boundaries
- What must NOT be changed?
- What refactors are forbidden?
- What rules are non-negotiable?

6.5 Execution plan
- What is the ordered sequence of steps/batches?
- What does each step produce?

6.6 Verification
- What tests/proofs are required?
- What output is expected?

6.7 Stop/Archive condition
- When is this pack obsolete?
- Where does it get archived?
- What artifact(s) become the new truth after completion?

6.8 Navigability
- If the pack is multi-file, does it include a Pack Index / Read Order file?

================================================================================
7) LIFECYCLE LAW (ACTIVE → OBSOLETE → ARCHIVED)
================================================================================

7.1 Creation
Execution Packs are created during or immediately after a Control Sync when the
need for deterministic execution is identified.

7.2 Activation must be explicit (anti-guess)
An Execution Pack is ACTIVE only when the current execution state explicitly names
it as binding for the current execution window.

Valid active-state sources include:
- a Control Sync decision captured in the Control Sync Snapshot
- the Bootstrap Prompt
- the Continuity Lock
- explicit Subject-state pointers naming the active pack

If no canonical artifact names an active pack, you MUST NOT assume one is active.

7.3 Active source and mirror rule
The canonical active source lives under:
- <Subject>_Data/Docs/Execution Packs/Active/

For rehydration convenience, a single pointer (or mirror metadata file) MAY exist under:
- <Subject>_Data/Latest Rehydration Pack/Execution Pack/

Rule:
- the pointer is convenience state
- the source pack is the execution artifact
- no Execution Pack may be fabricated merely to fill the Rehydration Pack

7.4 Refresh
An active Execution Pack is refreshed only when one of the following changes:
- source path changed
- source contents/signature changed
- day rollover requires a new active pointer file
- the bounded objective changed by lawful Control Sync decision

No-op refreshes must preserve the current active pack/pointer.

7.5 Completion
A pack is complete when its objective/deliverables are met and verification passes.

7.6 Archive
Execution Packs are consumable. They must be archived after completion or when superseded.

Archive rule:
- archive the source pack under:
  - <Subject>_Data/Docs/Execution Packs/Archived/
- archive the convenience pointer under:
  - <Subject>_Data/Archive/Latest Rehydration Pack/Execution Pack/

Archival is not deletion.
Archived packs remain for audit and pattern reuse.

7.7 Missing source rule
If an active Execution Pack pointer exists but the active source no longer exists:
- the pointer is invalid
- rehydration is incomplete
- repair is required before governed execution can continue

7.8 Ambiguity rule
If multiple active Execution Pack sources or multiple active pointers exist and no lawful pointer resolves them:
- STOP
- trigger Disclosure Gate
- repair the active state

7.9 “Packs do not create new system law” rule
An Execution Pack may constrain execution, but it must not silently create or modify
system law (Codex, Build Manual, governance canon, world-state legality).

If execution reveals a permanent rule or invariant:
- pause execution (or isolate the finding)
- run a Control Sync
- update the appropriate canonical artifact(s)
- then resume with an updated pack if needed

================================================================================
8) RELATIONSHIP TO OTHER ARTIFACTS (HOW IT FITS IN THE SYSTEM)
================================================================================

Execution Pack is part of the Synapse Guild Operating System (process), not the Guild body.

Canonical relationships:
- Control Sync produces decisions.
- Execution Pack converts those decisions into executable constraints + run plans.
- Guild Orders define the broader campaign objectives.
- Quests define atomic commitments.
- Dailies / governed execution uses the chosen scope.
- Snapshots record what happened and what is now true.
- Continuity Lock seals state for cross-chat resumption.
- Bootstrap Prompt initializes the next session’s stance to obey the above.

Execution Pack does NOT replace Guild Orders / Quests / Snapshots.
It constrains execution so those artifacts can be produced without drift.

================================================================================
9) REHYDRATION ROLE (HOW EXECUTION PACKS ARE USED ACROSS CHATS)
================================================================================

Execution Packs are part of rehydration only when one is active.

A Rehydration Pack is the minimum bundle required to resume work without drift.
Execution Pack’s role inside rehydration is:
- tell the new operator how the bounded work is currently constrained
- prevent re-deciding or improvising
- carry verification boundaries along with the work

Rehydration rule:
- If an active Execution Pack exists, the Rehydration Pack must expose it through the single active pointer.
- If no active Execution Pack exists, the Rehydration Pack must not invent one.

Typical rehydration bundle for governed execution:
- Continuity Lock (state + resume point)
- Bootstrap Prompt (stance + read order)
- Active Execution Pack pointer (if active)
- Latest relevant Snapshots
- Active Guild Orders (if a Raid is active)
- Accepted Quests (commitments)
- Relevant Codex / Build Manual / contracts

================================================================================
10) DISCRETION LAW (WHAT THE OPERATOR MAY DECIDE)
================================================================================

Execution Packs remove discretion in drift-sensitive areas, but they do not eliminate all discretion.
A pack must define where the operator may decide.

Allowed discretion (typical):
- minor implementation details that do not violate invariants
- formatting and ergonomics that do not change behavior
- test organization improvements that do not alter semantics
- batching output to avoid truncation

Forbidden discretion (typical):
- refactoring architecture when the pack says “no refactor”
- changing truth/immutability rules to “make it easier”
- bypassing consent or disclosure rules
- claiming work was done without verification artifacts/results
- silently overwriting durable artifacts
- changing module boundaries without governance decision

If a decision feels architectural, pause and trigger a Control Sync.

================================================================================
11) PACK STRUCTURE AND STORAGE (FILE SYSTEM LAW)
================================================================================

Execution Packs are subject-state artifacts.
They belong in <Subject>_Data (not in <Subject>_Engine and not in Governance).

Canonical storage:
- <Subject>_Data/Docs/Execution Packs/Active/
- <Subject>_Data/Docs/Execution Packs/Archived/

Optional convenience mirror:
- <Subject>_Data/Latest Rehydration Pack/Execution Pack/
- <Subject>_Data/Archive/Latest Rehydration Pack/Execution Pack/

Naming convention (recommended):
EXEC_PACK__<SCOPE>__<TOPIC>__YYYY-MM-DD__vX/
or for single-file packs:
EXEC_PACK__<SCOPE>__<TOPIC>__YYYY-MM-DD__vX.txt

Examples:
- EXEC_PACK__<SUBJECT>__BATCH_0_ENGINE_EXPANSION__2026-01-13__v1/
- EXEC_PACK__<SUBJECT>__V1_CONTRACTS_AND_PIPELINE__2026-01-13__v1/

Rule:
- Active packs must be obvious to find.
- Archived packs must be preserved, not edited.
- Active pointer files in the Rehydration Pack must resolve cleanly to active sources.

================================================================================
12) CASE STUDIES (WHY PACKS DIFFER)
================================================================================

These examples demonstrate that packs are situational by design.

12.1 Platform substrate pack
Trigger:
The Subject could not be built cleanly because platform primitives were missing.
Constraint:
No subject-specific code in core. Additive changes only. No router refactor.
Outcome:
Pack archived after primitives implemented and smoke tests passed.

12.2 Contracts pack
Trigger:
Long design conversation with high drift risk and many subtle invariants.
Constraint:
Artifact immutability, evidence/citations, indexing minimum, resume rules, adapter matrix.
Outcome:
Pack remains active until the governed window is complete; then archived.

12.3 Hypothetical design-to-implementation pack
Trigger:
Long Control Sync defining world rules, progression, and core constraints.
Constraint:
Forbid premature coding of features until rules are locked; preserve design invariants.
Outcome:
Pack archived when the phase is complete; a new pack may be authored for the next phase.

The difference is the point: packs solve drift for a specific execution window.

================================================================================
13) OPERATOR COMMANDS (CANONICAL PHRASES)
================================================================================

Once this law is accepted, the following phrases are valid commands:

- “Generate an Execution Pack for this Control Sync.”
Meaning:
Create the pack’s file(s), name them, and ensure the pack is navigable and complete.
If the pack is multi-file, produce a Pack Index / Read Order file first.
Ensure the pack satisfies MVEP before declaring it ready.

- “Activate this Execution Pack.”
Meaning:
Treat it as binding for the current work; obey constraints; produce required proofs.

- “Refresh the Execution Pack.”
Meaning:
A lawful change affected the active pack; update the source and active pointer without creating ambiguous parallel packs.

- “Archive this Execution Pack.”
Meaning:
The pack’s objective is done or superseded; move the source and pointer to archive; continue.

================================================================================
14) QUALITY BAR (WHAT MAKES A PACK “GOOD”)
================================================================================

A good Execution Pack makes execution boring.

You should be able to:
- resume after a week and not re-decide
- hand the pack to another operator and get consistent output
- prove what happened via tests/artifacts/snapshots
- know immediately whether a pack is active, missing, or obsolete
