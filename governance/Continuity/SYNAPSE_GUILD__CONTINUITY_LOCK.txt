================================================================================
0) PURPOSE OF THIS DOCUMENT
================================================================================

This document defines what a CONTINUITY LOCK is, why it exists, how it fits into
rehydration, and when it must be refreshed.

After reading this document, the phrase “Refresh the Continuity Lock” MUST be
sufficient instruction for any compliant operator (human or AI).

A Continuity Lock exists to solve one problem:

> How do we continue a complex project and its consequences when the chat itself
> cannot be carried forward?

Chats end. Memory decays. Context drifts. Decisions get re-argued.
Continuity Locks prevent that.

Governance note (anti-confusion):
- This file defines Continuity Lock law and quality bar.
- Active Continuity Locks are subject-state artifacts stored in <Subject>_Data.
- This law does NOT live inside the active Subject Pack. It governs it.

================================================================================
1) CORE DEFINITION (CANON)
================================================================================

1.1 Continuity Lock — definition
A Continuity Lock is a state-sealing governance artifact that preserves continuity across sessions
by explicitly declaring what is true, binding, irrelevant, and next, WITHOUT relying on chat history.

One-sentence definition:
A Continuity Lock preserves project continuity without chat history by sealing state, decisions,
scope boundaries, and the resume point.

1.2 Authority
- A Continuity Lock is authoritative state.
- Conversation is input only.
- If the Lock and conversation conflict, the Lock wins.

Truth Gate note:
- A Continuity Lock “exists” only if the file exists on disk in canonical Subject_Data.
- A Continuity Lock written only in chat is NON-AUTHORITATIVE.

================================================================================
2) WHAT PROBLEM IT SOLVES (WHY IT EXISTS)
================================================================================

Continuity Locks exist because:
- AI chat sessions have finite length.
- Chat history is not guaranteed to persist or be readable.
- Humans and AI forget *why* decisions were made.
- Re-explaining context causes drift, re-litigation, and accidental redesign.

Without a Continuity Lock, new sessions tend to:
- revisit rejected ideas
- reinterpret settled decisions
- wander into new scope unintentionally
- lose momentum and mode

A Continuity Lock prevents all of this by acting as a state seal.

================================================================================
3) WHAT A CONTINUITY LOCK IS — AND IS NOT
================================================================================

3.1 What it IS
A Continuity Lock is:
- a state declaration
- a boundary marker
- a resume instruction
- a compression of consequence (not a compression of conversation)

3.2 What it is NOT
A Continuity Lock is NOT:
- a transcript
- a summary of everything discussed
- a brainstorming log
- a task list
- an execution plan
- a Bootstrap Prompt

It does not preserve conversation.
It preserves consequence.

================================================================================
4) REQUIRED CONTENTS (VALIDITY REQUIREMENTS)
================================================================================

A valid Continuity Lock MUST explicitly state the following.
If any required section is missing or ambiguous, continuity is degraded and rehydration is INVALID
until repaired.

4.1 Subject identity
- Subject key / name (the exact canonical identifier)

4.2 World state
- Fog of War or Fog Lifted
- If Fog Lifted, the Lock MUST acknowledge Codex Freeze exists (CODEX_FREEZE.md)

4.3 Current phase
- Where we are right now
- Example: “Ambient drafting active; governed execution still gated”

4.4 Binding decisions (LAW)
- Decisions that are now binding
- Constraints that MUST NOT be re-litigated casually

4.5 Deferred / rejected / unknown items
- Explicitly list:
  - rejected ideas (NO)
  - deferred ideas (NOT NOW)
  - open questions (UNKNOWN)

4.6 Active scope and commitments
- What is active now
- What is paused
- What is complete
- If an Execution Pack is currently binding, it MUST be named here (by path or explicit pointer)

4.7 Required reading for the next session
- Which snapshots / packs / documents MUST be read first

4.8 Resume point (the next deterministic action)
- The exact next action or question the next session MUST address

Rule:
If it matters later, it MUST be written into the Continuity Lock or a Snapshot.

================================================================================
5) EXPLICIT STATE VS IMPLICIT HISTORY
================================================================================

A Continuity Lock does not carry history forward.
It carries the implications of that history forward.

Discussion and exploration are allowed to decay unless they are promoted into one of:
- a binding decision
- a constraint
- a rejection or deferral record
- a resume instruction
- an active execution commitment

================================================================================
6) RELATIONSHIP TO SNAPSHOTS
================================================================================

Continuity Locks and Snapshots are complementary.

Snapshots:
- record what happened
- preserve rationale
- provide auditability

Continuity Locks:
- declare what is now binding
- declare what no longer matters
- tell the next session how to resume

Typical pattern:
- Control Sync Snapshot records decisions and reasoning
- Continuity Lock seals the binding consequence + resume point

A Continuity Lock MAY reference Snapshots by explicit path.
If referenced, the Lock MUST list the Snapshot filename(s) and storage location(s).
It does NOT replace them.

================================================================================
7) RELATIONSHIP TO EXECUTION PACKS
================================================================================

Execution Packs and Continuity Locks solve different problems.

Execution Packs:
- constrain how work is executed
- prevent implementation drift

Continuity Locks:
- constrain interpretation
- prevent decision drift

A Continuity Lock MAY state:
- “An Execution Pack exists and is binding”
- “Execution Pack X MUST be read before proceeding”

Reference rule:
- If the Lock references an Execution Pack, it MUST name the pack and its canonical location (path or explicit pointer).
- If the Lock does not reference an Execution Pack, no operator may assume one is active.

But the Lock itself MUST NOT contain step-by-step execution instructions.

================================================================================
8) RELATIONSHIP TO BOOTSTRAP PROMPTS
================================================================================

Continuity Locks and Bootstrap Prompts are often used together, but they are not the same.

Continuity Lock answers:
- “What is true?”
- “What is binding?”
- “Where are we resuming?”

Bootstrap Prompt answers:
- “How should the AI behave now?”
- “What role is it in?”
- “What should it do first?”

Rehydration ordering clarification:
- A Bootstrap Prompt MAY be read first to establish stance and protocol.
- A Continuity Lock MUST be applied before planning, quest acceptance, or execution.
- If Bootstrap Prompt and Continuity Lock conflict on state, the Continuity Lock wins.

================================================================================
9) WHEN A CONTINUITY LOCK IS REQUIRED
================================================================================

A Continuity Lock MUST exist whenever a session intends to resume deterministically.

A Continuity Lock MUST be created or refreshed when any of the following are true:
- a Control Sync concluded (especially if binding decisions were made)
- execution depends on preserved context
- a mode or direction MUST NOT change
- work will resume later (default for continuing subjects)
- a session is nearing its limit AND the next session must continue without re-litigation
- a binding decision was logged
- a disclosure changed what is safe or legal
- the active run changed materially
- the current accepted Quest changed
- a formalization changed the active truth surface
- verification changed what the system may claim
- Codex / Build Manual state changed in a way that affects execution
- a day boundary or end-of-day seal changes the active resume surface

Omission rule (strict):
- If Brains explicitly declares “No continuity required,” a Continuity Lock may be omitted.
- If a Continuity Lock is omitted, the next session MUST NOT claim deterministic rehydration.

Operational consequence:
- If a continuing Subject is missing a Latest Continuity Lock, the Rehydration Pack is INCOMPLETE.

================================================================================
10) STORAGE + NAMING (DETERMINISTIC)
================================================================================

Rule: The Governance Pack stays stateless.
Do not store subject-specific locks inside the Governance Pack.

Canonical active location:
- <Subject>_Data/Latest Rehydration Pack/

Canonical archive location:
- <Subject>_Data/Archive/Latest Rehydration Pack/

Naming requirements:
- Filename MUST contain: CONTINUITY_LOCK
- Filename MUST include a date token: YYYY-MM-DD
- Same-day material refreshes MAY use deterministic revision suffixes (for example: __02)

Authoritative resolution rule:
- The active Continuity Lock is resolved by explicit Subject-state pointers first.
- Filename/date rules support deterministic storage and archival.
- Filename/date rules do NOT overrule explicit active pointers.

Ambiguity rule:
- If multiple candidate active locks exist and cannot be deterministically resolved, STOP.
  Trigger Disclosure Gate and repair naming/state.

Supersession rule:
- When a new Continuity Lock supersedes an old one:
  - archive the old lock
  - activate the new lock
  - update Subject-state pointers to the single active lock

================================================================================
11) MINIMUM FORMAT (NOT OPTIONAL)
================================================================================

A Continuity Lock MUST contain these headings (verbatim or clearly equivalent):

- SUBJECT:
- DATE:
- WORLD STATE:
- CURRENT PHASE:
- BINDING DECISIONS (LAW):
- DEFERRED / REJECTED / UNKNOWN:
- ACTIVE SCOPE / COMMITMENTS:
- REQUIRED READ FIRST:
- RESUME POINT:

Quality note:
- “BINDING DECISIONS” MUST be explicit and written as law (no hedging language).

Minimum content requirements (by heading):
- SUBJECT: Subject key/name. MUST match the <Subject>_Data root folder name.
- DATE: YYYY-MM-DD.
- WORLD STATE: FOG OF WAR or FOG LIFTED.
  - If FOG LIFTED, the Lock MUST reference the Codex Freeze marker (CODEX_FREEZE.md) as the legal basis.
- CURRENT PHASE: One sentence naming the current operating phase and what materially changed since last lock.
- BINDING DECISIONS (LAW): Bullet list of MUST/MUST NOT statements. No hedging.
- DEFERRED / REJECTED / UNKNOWN: Explicit list. If none, write NONE.
- ACTIVE SCOPE / COMMITMENTS: What is actively in-flight.
  - If a Raid/Guild Orders are active: name them.
  - If any Quests are ACCEPTED: list Quest IDs (or filenames) explicitly.
  - If an Execution Pack is binding: list its explicit path or active pointer.
- REQUIRED READ FIRST: Exact artifacts the next session MUST read before any planning/execution (paths or names).
- RESUME POINT: The first artifact to open + the next action to take.
  - If Quests are ACCEPTED, specify the next Quest to execute.

================================================================================
12) QUALITY BAR (WHAT MAKES A GOOD LOCK)
================================================================================

A good Continuity Lock:
- makes the next session boring
- prevents re-argument
- prevents scope creep
- allows seamless continuation without anxiety
- refreshes when truth changed and stays stable when truth did not

A bad Continuity Lock:
- is vague
- reads like a summary
- does not state what is irrelevant
- does not specify next action
- rewrites itself on every no-op state check

================================================================================
13) COMMON FAILURE MODES (AND HOW TO AVOID THEM)
================================================================================

Failure: Treating the lock like a summary
Fix: Declare state and decisions, not discussion.

Failure: Carrying everything forward “just in case”
Fix: Explicitly mark what is deferred or irrelevant.

Failure: No clear resume point
Fix: End the lock with “RESUME POINT: …”

Failure: Multiple competing “latest” locks
Fix: Archive superseded locks and enforce pointer-based active resolution.

Failure: Rewriting the lock on every transition tick
Fix: Refresh only when truth materially changed.

================================================================================
14) CANONICAL COMMANDS
================================================================================

Once this law is accepted, the following phrases are valid:

- “Create a Continuity Lock.”
Meaning:
Seal the current state so the next session can resume without chat history.

- “Refresh the Continuity Lock.”
Meaning:
State changed materially; either preserve the current lock if nothing changed or create a new lock and archive the old one.

- “Rehydrate from this Continuity Lock.”
Meaning:
Treat it as authoritative state and proceed accordingly.
