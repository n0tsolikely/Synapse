================================================================================

This document defines what the TALENT TREE is, why it exists, and how it is used.

After reading this document, the phrase “Update the Talent Tree” MUST be
sufficient instruction for any compliant operator (human or AI).

The Talent Tree is not a task list, not a backlog, and not a reward system.
It is the identity ledger of the system.

This is governance law.

================================================================================
1) CORE DEFINITION (CANON)
================================================================================

1.1 Talent Tree — definition
The Talent Tree is the authoritative, human-readable ledger of the system’s
currently unlocked capabilities.

Each Talent represents a capability that now exists and MUST be respected
by all future work unless explicitly reversed via respec.

One-sentence definition:
The Talent Tree records what the system can now do, and therefore what future
work MUST assume is true.

================================================================================
2) WHAT PROBLEM THE TALENT TREE SOLVES
================================================================================

The Talent Tree exists because:

- Systems evolve over time.
- Capabilities accumulate.
- Without an explicit identity ledger, teams forget what is already possible.
- Drift occurs when future work ignores or re-implements existing capability.

Without a Talent Tree:
- features get duplicated
- constraints are violated accidentally
- scope resets silently
- long-term identity becomes fuzzy

The Talent Tree prevents identity drift.

================================================================================
3) WHAT THE TALENT TREE IS — AND IS NOT
================================================================================

3.1 What the Talent Tree IS
The Talent Tree is:
- an identity ledger
- an authoritative statement of current capability
- forward-looking (it constrains future work)
- append-only (corrections are appended, not rewritten)

3.2 What the Talent Tree is NOT
The Talent Tree is NOT:
- a roadmap
- a backlog
- a to-do list
- a Quest list
- a reward or gamification system
- a replacement for Guild Orders

Talents describe what is now true, not what we plan to do next.

================================================================================
4) STRUCTURE: THE THREE CANONICAL FILES
================================================================================

The Talent system consists of exactly three subject-state files.

Canonical location (per Subject):

  <Subject>_Data/Talent Tree/

Files:
- TALENT_TREE.txt
- TALENT_LOG.txt
- RESPEC_RULES.txt

No additional files are created for individual Talents.
Identity must remain readable at a glance.

--------------------------------------------------------------------------------
4.1 TALENT_TREE.txt — Current identity (subject-state)
--------------------------------------------------------------------------------

Purpose:
- Human-readable statement of unlocked capabilities.
- Represents the current identity of the system.

Rules:
- Append new Talent entries at the bottom.
- Do not delete old entries.
- Corrections are appended, not rewritten.
- Each Talent entry MUST be explicit about its implications.

--------------------------------------------------------------------------------
4.2 TALENT_LOG.txt — Audit history (subject-state)
--------------------------------------------------------------------------------

Purpose:
- Append-only history of Talent Points earned and spent.
- Records which Quest caused which Talent to be added (or respec’d).

Rules:
- Never delete entries.
- Corrections are appended, not rewritten.

--------------------------------------------------------------------------------
4.3 RESPEC_RULES.txt — Governance escape hatch (subject-state)
--------------------------------------------------------------------------------

Purpose:
- Defines the ceremonial process for reversing or changing a Talent.

Rules:
- Respec is rare and deliberate.
- Must occur during a Control Sync.
- Must be logged explicitly in TALENT_LOG.

--------------------------------------------------------------------------------
4.4 Initialization rule (missing files)
--------------------------------------------------------------------------------

If any of the three canonical files are missing from:

  <Subject>_Data/Talent Tree/

the Talent Tree is considered UNINITIALIZED for that Subject.

Operator response:
- Create the missing files by copying the templates from the Governance Pack
  (this folder).
- Record the initialization in the next Snapshot.

================================================================================
5) TALENTS AND QUESTS (CRITICAL RELATIONSHIP)
================================================================================

Talents are earned through Quests — but not all Quests.

5.1 Talent-awarding Quests
A Quest awards a Talent Point only if it:
- introduces a new capability, OR
- meaningfully expands an existing capability

Examples:
- adding a new pipeline stage
- enabling search
- adding a new interface door
- enabling evidence-backed outputs

5.2 Non-talent Quests
Quests that typically do NOT award Talent Points:
- bug fixes
- refactors
- cleanup work
- documentation updates
- renaming or reorganizing files
- maintenance tasks

The Quest MUST explicitly state whether it awards a Talent Point.
No assumption is allowed.

================================================================================
6) HOW TALENTS CONSTRAIN FUTURE WORK
================================================================================

Talents are forward constraints.

Once a Talent exists:
- future Guild Orders MUST respect it
- future Quests MUST assume it exists
- future Execution Packs MUST NOT violate it

Example:
If a Talent states “Artifacts are immutable,”
then no future work may silently overwrite artifacts.

This is why Talents are powerful.
They encode non-negotiable reality.

================================================================================
7) UPDATING THE TALENT TREE (WHEN AND HOW)
================================================================================

7.1 When to update the Talent Tree
Update the Talent Tree only when:
- a Talent Point is awarded AND
- the new capability is confirmed COMPLETE AND
- the Quest completion criteria are met AND VERIFIED

Do NOT update the Talent Tree:
- mid-Quest
- based on intent or plans
- for work that does not change capability

If verification did not occur, the Talent update is illegal.

7.2 How the update happens (deterministic)
Typical flow:
1) Quest is completed and verified (Truth Gate compliant receipts exist).
2) Quest awards a Talent Point (explicit YES in the Quest artifact).
3) Append a new Talent entry to <Subject>_Data/Talent Tree/TALENT_TREE.txt.
4) Append a log entry to <Subject>_Data/Talent Tree/TALENT_LOG.txt.
5) Mention the Talent update in the End-of-Day Snapshot.

================================================================================
8) TALENT FORMAT (CANONICAL)
================================================================================

Each Talent entry MUST include:

- TALENT ID
- Name
- Unlocked On (date)
- Source Quest
- Scope (<SUBJECT> / <SUBJECT> / System / Other)
- Description (what capability now exists)
- Implications (what future work must respect)
- Notes (optional)

This format is defined in TALENT_TREE.txt and must not drift.

================================================================================
9) RESPEC (CHANGING HISTORY WITHOUT LYING)
================================================================================

Reality changes. Mistakes happen. Architecture evolves.

Respec exists so the system can change without rewriting history.

Rules (from RESPEC_RULES.txt):
- Respec is rare.
- Respec requires a Control Sync.
- Respec MUST state:
  - what Talent changed
  - why
  - consequences
  - migration plan
- Respec MUST be logged in TALENT_LOG.

Respec is intentionally painful.
That pain prevents casual history edits.

================================================================================
10) TALENT TREE VS OTHER ARTIFACTS
================================================================================

Canonical placement (conceptual):

- Guild Orders:
  Define what we intend to build.

- Quests:
  Define what we execute.

- Talent Tree:
  Defines what we have become capable of.

- Snapshots:
  Record what happened.

The Talent Tree is not replaced by any of these.
It answers a different question.

================================================================================
11) QUALITY BAR (WHAT MAKES A GOOD TALENT)
================================================================================

A good Talent:
- describes a real capability
- is specific and testable
- creates meaningful future constraints
- can be understood in isolation

A bad Talent:
- is vague (“system improved”)
- duplicates existing Talents
- describes effort instead of capability
- has no implications

If a Talent cannot clearly state its implications, it is not ready.

================================================================================
12) CANONICAL COMMANDS
================================================================================

Once this law is accepted, the following phrases are valid:

- “Award a Talent Point.”
Meaning:
A completed Quest introduced or expanded a capability.

- “Update the Talent Tree.”
Meaning:
Append a new Talent entry describing the unlocked capability AND log it.

- “Respec this Talent.”
Meaning:
Initiate the ceremonial respec process during a Control Sync.
