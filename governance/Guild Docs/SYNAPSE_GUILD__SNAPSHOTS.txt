================================================================================
0) PURPOSE OF THIS DOCUMENT
================================================================================

This document defines what SNAPSHOTS are, why they exist, and how they are used.

After reading this document, the phrase “Write a Snapshot” MUST be sufficient
instruction for any compliant operator (human or AI).

Snapshots are the ground‑truth memory system of the Synapse Guild.
They preserve reality over time without relying on chat history or human recall.

Snapshots are governance law.

================================================================================
1) CORE DEFINITION (CANON)
================================================================================

1.1 Snapshot — definition
A Snapshot is a factual, point‑in‑time record of what is true at a specific moment.

A Snapshot answers one question:

> “What is now true, and how did we get here?”

One‑sentence definition:
A Snapshot is an immutable record of reality captured at a specific moment in time.

1.2 Snapshot Artifact — definition
A Snapshot is an artifact file stored in the Subject’s data directory.

A Snapshot is not “what the operator remembers.”
A Snapshot is not “what was said in chat.”
A Snapshot is a file that exists in <Subject>_Data/Snapshots/.

If the Snapshot file does not exist, then the Snapshot does not exist.

================================================================================
2) WHAT PROBLEM SNAPSHOTS SOLVE
================================================================================

Snapshots exist because:

- Memory drifts.
- Chats end.
- Humans forget details.
- Systems evolve incrementally.

Without Snapshots:
- history gets rewritten unintentionally
- decisions lose context
- execution becomes non‑auditable
- continuity breaks across days, weeks, or months

Snapshots prevent historical drift.

================================================================================
3) WHAT SNAPSHOTS ARE — AND ARE NOT
================================================================================

3.1 What Snapshots ARE
Snapshots are:
- factual
- time‑bound
- immutable
- evidence‑based

3.2 What Snapshots are NOT
Snapshots are NOT:
- plans
- intentions
- promises
- speculative notes
- rewritten after the fact

If something is not true yet, it does not belong in a Snapshot.

3.3 Immutability rule
Snapshots MUST NOT be edited to change historical truth.

If a Snapshot is wrong or missing context:
- write a new Snapshot that corrects it
- reference the prior Snapshot explicitly
- do not rewrite history

================================================================================
4) THE THREE SNAPSHOT TYPES (CANON)
================================================================================

There are exactly three Snapshot types.

Each exists for a different purpose.
They MUST NOT be conflated.

--------------------------------------------------------------------------------
4.1 CONTROL SYNC SNAPSHOT
--------------------------------------------------------------------------------

Purpose:
Capture binding decisions produced by a Control Sync.

Mandatory trigger:
A Control Sync MUST end with exactly one Snapshot:
- Control Sync Snapshot if any binding decision was made
- General Snapshot if no binding decisions were made (alignment/context only)

Used when binding decisions occur, including (non‑exhaustive):
- scope was defined or changed
- Guild Orders were created or updated
- Dungeons/Quests were created, accepted, rejected, re‑scoped
- Execution Packs or canon documents were generated
- authoritative decisions were made

What it records (minimum):
- decisions made
- what is now binding
- what is explicitly NOT decided
- references to new or updated artifacts (paths)
- the declared scope commitment for the session/day (if applicable)

What it does NOT record:
- exploratory discussion
- brainstorming
- undecided ideas

Think of it as:
“Here is what came out of the meeting that now counts.”

--------------------------------------------------------------------------------
4.2 END‑OF‑DAY (EOD) SNAPSHOT
--------------------------------------------------------------------------------

Purpose:
Capture what actually happened during execution.

Mandatory trigger:
When Hands ends a work session (“stop”, “done”, “end session”, or equivalent),
Brains MUST write an End‑of‑Day Snapshot.

If no execution occurred:
- the End‑of‑Day Snapshot MUST explicitly state that no execution occurred
- it MUST reference the latest Control Sync Snapshot (if one occurred)

Used when:
- work was done
- Quests progressed or completed
- files or code changed
- blockers or failures were discovered

What it records (minimum):
- what was worked on (by Quest and by concrete work performed)
- what was completed
- what is now true in the system
- what remains incomplete (and why)
- any problems uncovered (errors, failures, blockers)
- where the raw receipts live (Execution Audits, logs, test output files) if applicable

What it does NOT record:
- future plans
- decisions that were not made
- speculation

Think of it as:
“If everything stopped now, this is the state we would resume from.”

--------------------------------------------------------------------------------
4.3 GENERAL SNAPSHOT
--------------------------------------------------------------------------------

Purpose:
Capture any important moment that does not fit cleanly into the other two types.

Mandatory trigger (alignment‑only Control Sync):
If a Control Sync ended with no binding decisions, the Control Sync MUST end with
exactly one General Snapshot.

Used when:
- a Control Sync was discussion/alignment/context only
- context was rehydrated after time away
- insight or understanding emerged
- governance or philosophy was clarified
- an important moment occurred mid‑day

What it records (minimum):
- context
- understanding
- rationale
- observations worth preserving

What it does NOT record:
- binding decisions (unless explicitly stated as binding)
- execution logs

Think of it as:
“This moment mattered. Future‑us should know it happened.”

================================================================================
5) SNAPSHOTS AND CONTROL SYNCS
================================================================================

Control Syncs are moments.
Snapshots are records.

A Control Sync itself is ephemeral.
Its outcomes are not.

Rules (hard):
- A Control Sync MUST end with exactly one Snapshot.
  - Control Sync Snapshot if binding decisions exist.
  - General Snapshot if no binding decisions exist.
- Binding decisions MUST NOT exist only in chat.
  If a decision is binding, it MUST be recorded in a Snapshot.

If a Snapshot cannot be written (missing Subject, missing permissions, environment limits):
- Disclosure Gate MUST trigger.
- The Control Sync is not considered closed.
- Execution authority MUST NOT advance.

================================================================================
6) SNAPSHOTS AND DAILIES
================================================================================

Snapshots are a mandatory component of the Dailies loop.

In Dailies:
- Work is executed.
- Reality changes.
- A Snapshot records that change.

Rules (hard):
- The Dailies loop MUST include:
  - a Snapshot at the end of the Control Sync (Control Sync Snapshot or General Snapshot per Section 5)
  - an End‑of‑Day Snapshot when Hands ends the work session

Execution without Snapshots causes drift.
Drift invalidates continuity.

================================================================================
7) SNAPSHOTS AND CONTINUITY LOCKS
================================================================================

Snapshots and Continuity Locks serve different roles.

- Snapshots preserve history.
- Continuity Locks preserve current state.

Snapshots accumulate.
Continuity Locks replace one another.

A Continuity Lock may reference Snapshots.
Snapshots MUST NEVER be rewritten to match a Continuity Lock.

================================================================================
8) FORMAT, STORAGE, AND NAMING (CANON)
================================================================================

Snapshots are text files stored inside the Subject data folder.

<Subject>_Data/Snapshots/
  General/
  Control Sync/
  End of Day/

Canonical rules:
- Snapshots belong to the Subject, not to Governance.
- Governance defines the law; <Subject>_Data stores the artifacts.
- If the required Snapshot directories do not exist, Brains MUST create them
  (or trigger Disclosure Gate if creation is not possible).

----------------------------------------
8.1 STORAGE LOCATIONS (CANON)
----------------------------------------
- General Snapshots:
  <Subject>_Data/Snapshots/General/

- Control Sync Snapshots:
  <Subject>_Data/Snapshots/Control Sync/

- End of Day Snapshots:
  <Subject>_Data/Snapshots/End of Day/

----------------------------------------
8.2 NAMING CONVENTION (CANON)
----------------------------------------
Snapshot filenames MUST make type + subject + date obvious.

Canonical pattern:
<Subject>_<SnapshotType>_Snapshot_YYYY-MM-DD.txt

Where <SnapshotType> is one of:
- Control_Sync
- End_of_Day
- General

Examples:
- STUART_Control_Sync_Snapshot_2026-02-02.txt
- STUART_End_of_Day_Snapshot_2026-02-02.txt
- STUART_General_Snapshot_2026-02-02.txt

Strict filename termination rule (hard):
- Snapshot filenames MUST end with the date token: YYYY-MM-DD.txt
- No suffix-after-date is allowed (no _topic, no _notes, etc).
- Only allowed insertion is SAME-DAY DUPLICATE COUNTER (see below), which MUST appear BEFORE the date token.

Duplicate same-day rule (when multiple Snapshots of the same type are required on the same date):
- First snapshot of the day:
  - <Subject>_<SnapshotType>_Snapshot_YYYY-MM-DD.txt
- Second snapshot of the day (counter is lexicographic-safe and does NOT appear after the date):
  - <Subject>_<SnapshotType>_Snapshot__02_YYYY-MM-DD.txt
- Third snapshot of the day:
  - <Subject>_<SnapshotType>_Snapshot__03_YYYY-MM-DD.txt
- Counters MUST be zero-padded (02, 03, …) for deterministic sorting.

----------------------------------------
8.3 REHYDRATION ORDER (LOCK)
----------------------------------------
When resuming a Subject, the ordering is locked:

1) Review the latest Control Sync Snapshot
  (intent / scope / decisions)

2) Review the latest End of Day Snapshot
  (execution reality / outcomes / blockers)

This prevents intent‑reality inversion during rehydration.

================================================================================
9) QUALITY BAR (WHAT MAKES A GOOD SNAPSHOT)
================================================================================

A good Snapshot:
- is honest
- is explicit
- separates fact from interpretation
- can be trusted months later
- is extremely detailed and human readable
- able for a guild member to read who was not present that day and not miss a beat

A bad Snapshot:
- mixes intent with reality
- omits important outcomes
- rewrites history
- relies on memory instead of evidence
- omits to much information
- leaves ambiguity in elements it discusses and states

Snapshots must be boring and reliable.

--------------------------------------------------------------------------------
9.1 MINIMUM STRUCTURE (TEMPLATES ARE MINIMUMS, NOT LIMITS)
--------------------------------------------------------------------------------

Snapshots MUST capture enough truth that tomorrow’s Control Sync can resume without guessing.

Canonical minimum Snapshot templates live here:
- Guild Docs/SYNAPSE_GUILD__SNAPSHOT_TEMPLATES.txt

Rules (hard):
- Templates define minimum sections.
- The operator MUST include any additional relevant information even if the template
  does not list it (bugs, failures, blockers, weird behavior, decisions, scope changes).

End‑of‑Day Snapshot quality bar (hard):
- The Snapshot MUST describe work performed in concrete terms.
  It is NOT sufficient to only list Quest IDs/titles.
- For each Quest worked on, the Snapshot MUST capture (minimum):
  - what changed (conceptually)
  - what actions were taken (high level steps)
  - what the outcome was
  - what is left (if incomplete) and why
- If anything failed, the Snapshot MUST record:
  - what failed
  - why it failed (blocking reason)
  - the relevant error output location (Execution Audit / log / receipt path)
  - the next action required to avoid repeating the failure

================================================================================
10) CANONICAL COMMANDS
================================================================================

Once this law is accepted, the following phrases are valid:

- “Write a Snapshot.”
Meaning:
Record the current reality truthfully in the correct Snapshot type.

- “Write a Control Sync Snapshot.”
Meaning:
Record binding decisions and artifact updates produced by the Control Sync.

- “Write an End‑of‑Day Snapshot.”
Meaning:
Record what actually happened during execution and what is now true.

- “This needs a Snapshot.”
Meaning:
What just happened MUST be preserved as historical truth.

- “Reference this Snapshot.”
Meaning:
Use it as evidence of what occurred.
