# SYNAPSE GUILD — CODEX (v1.5)
Last Updated: 2026-03-13
================================================================================
0) DEFINITION
================================================================================

The Codex is the authoritative map of a Subject.
It describes what exists, how it behaves, where it lives, and what is allowed.

Aliases (routing, non-optional):
- Codex = Map
- TOC = Legend

Purpose:
- Eliminate ambiguity around what the Subject IS.
- Preserve the Subject so it can be rebuilt, resumed, or reasoned about without guessing.
- Give execution a stable constitutional spine once that spine is ready to be frozen.

Non-negotiable rule:
- Conversation is INPUT ONLY.
- The Codex (and other Subject_Data artifacts) are AUTHORITY.

The Codex is NOT the same as:
- Build Manual (HOW to make the Codex true)
- Talent Tree (what capabilities are known to exist now)
- Quest Board (what work is proposed / formalized / accepted)
- REHYDRATE / VISION surfaces (compressed current-state orientation)

================================================================================
1) WORLD STATE + CODEX MATURITY
================================================================================

Default state for new Subjects is Fog of War.

While Fog of War is active:
- Discovery, drafting, and structural clarification are allowed.
- The TOC and Codex may be drafted.
- The Codex may grow section-by-section alongside the work.
- Ambient quest detection, candidate drafting, and BOARD drafting may occur where lawful.
- Governed execution is forbidden.

Fog is lifted ONLY by Codex Freeze.
Gate condition (must both be true):
1) Brains explicitly declares the Codex frozen, AND
2) the marker file exists:
   - <Subject>_Data/Codex/CODEX_FREEZE.md

If the marker file does not exist, the system MUST assume World State remains Fog of War.

A TOC alone does not authorize governed execution.
A draft Codex alone does not authorize governed execution.
Only Codex Freeze does.

Important clarification:
- Codex Freeze does NOT require the Codex to become a perfect encyclopedia before any serious movement can occur.
- Codex Freeze requires that the Codex spine is stable enough that governed execution will not constantly invalidate itself.
- The Codex may continue to evolve after Freeze, but structural meaning changes MUST be deliberate and recorded.

Recommended maturity language while Fog remains active:
- Codex Seed: enough truth to begin orienting the Subject
- Codex Draft: multiple sections exist, but the Subject is still being discovered
- Codex Spine: core structure and governing meaning are stable enough to freeze once approved
- Codex Frozen: governed execution is legal

================================================================================
2) CODEX ARTIFACTS (STRUCTURE + STORAGE)
================================================================================

Codex artifacts live in the Subject’s Data pack.
Canonical folder shape is defined in:
- Processes/SYNAPSE_GUILD__SUBJECT_INITIALIZATION_AND_INCUBATION.txt

Minimum expected Codex artifacts:
- <Subject>_Data/Codex/TOC.md
  - During incubation, TOC_DRAFT.md is permitted.
- <Subject>_Data/Codex/Sections/
  - one subsection per file
  - section file names MUST be numbered to mirror TOC numbering
- <Subject>_Data/Codex/<Subject>_Codex_Full.txt
  - deterministic stitch of TOC + all section files
- <Subject>_Data/Codex/CODEX_FREEZE.md
  - exists only after Brains declares Codex Freeze

Section numbering law:
- TOC numbering is the canonical reference ID.
- A section file’s numbering MUST match its TOC entry numbering.
- If numbering changes, it is a structural change (must be recorded in Control Sync Snapshot).

================================================================================
3) WHAT THE CODEX CONTAINS
================================================================================

A complete Codex MUST cover the Subject end-to-end (domain-appropriate).
At minimum, the Codex MUST contain:
- Purpose + scope (what it is)
- Constraints (what it must obey)
- Concepts and definitions (shared meanings)
- Capabilities / features (what it does)
- Workflows (how it behaves)
- Boundaries (what it is NOT)

Domain additions (examples):
- programs: data model, interfaces, architecture, operational procedures
- books: tone, POV, character rules, world rules, plot structure, style constraints
- games: mechanics, progression, economy rules, content structure

The Codex MUST explicitly distinguish:
- MUST / required
- MAY / optional
- DEFERRED (explicitly deferred; not hand-waved)

The Codex is written as if the Subject already exists.
That means it records the Subject’s governing truth, not the operator’s temporary uncertainty.
If truth is still unknown, keep it in Incubation / Discovery surfaces until it can be stated without lying.

================================================================================
4) CODEX PURITY RULE (NOT AN EXECUTION PLAN)
================================================================================

The Codex describes what the Subject IS, as if it already exists.

The Codex MUST NOT contain operational execution artifacts such as:
- Guild Orders / Raids
- Quests / Quest Board content
- Dailies procedures
- Audit logs
- Snapshots

Those belong in Subject Data (examples):
- <Subject>_Data/Guild Orders/
- <Subject>_Data/Quest Board/
- <Subject>_Data/Audits/
- <Subject>_Data/Snapshots/

The Codex is also NOT the Build Manual.
- The Codex states WHAT is true.
- The Build Manual states HOW to make or maintain that truth without redefining it.

The Codex is also NOT the Talent Tree.
- The Codex may describe intended and governing capabilities.
- The Talent Tree records currently evidenced capability unlocks and durable capability growth.

A concise VISION surface MAY exist for fast orientation.
That surface is subordinate to the Codex and MUST NOT replace it.

================================================================================
5) CODEX QUALITY BAR (NO GUESSING)
================================================================================

The Codex must be written “as if the system already exists.”

Hard prohibitions:
- No undefined placeholders (e.g., TBD, ???).
- No hand-waving (“we’ll do it later”) inside canonical Codex sections.
- No magic or implicit behavior.

If something is not decided:
- keep it in Incubation as Discoveries
- do NOT freeze the Codex

Definition law:
- Any term used as a rule anchor MUST be defined in the Codex (or explicitly referenced to a Codex definition section).

================================================================================
6) HOW THE CODEX IS BUILT (DETERMINISTIC PIPELINE)
================================================================================

Codex formation is governed by The Loop:
- Incubation → TOC → Sections → Stitch → Freeze

Key rule:
- The TOC MUST be derived from discovered and captured decisions, not guesses.

Clarification:
- In the ambient sidecar model, Codex growth may proceed alongside lawful drafting, candidate detection, and exploratory work.
- This does NOT make governed execution legal.
- It means the Codex is allowed to mature while the Subject is still under Fog.

When the Codex is large, use:
- Processes/SYNAPSE_GUILD__LARGE_ARTIFACT_COMPOSITION.txt

--------------------------------------------------------------------------------
6.1 The TOC (Legend) build rule
--------------------------------------------------------------------------------

The TOC is the navigational contract for Codex construction.

TOC MUST:
- have stable numbering
- have clear, non-overlapping scopes per entry
- be complete enough that “write section X” is unambiguous

TOC MUST NOT:
- imply governed execution readiness by itself
- imply completeness of decisions by itself

--------------------------------------------------------------------------------
6.2 Section-by-section build loop (inner loop)
--------------------------------------------------------------------------------

Codex sections MUST be written one-by-one.
Each section file is the atomic unit of Codex construction.

For each TOC entry:
1) WRITE the section file
   - must match TOC numbering + title
   - must fully cover the entry’s scope
   - must define rules and constraints without hand-waving
2) SECTION AUDIT (mandatory)
   - verify numbering matches TOC
   - verify section stays within its scope (no leakage)
   - verify no placeholders / undefined anchors
   - verify terms used as anchors are defined (or explicitly reference a definition section)
3) PROGRESS MARK
   - progress is proven by the presence of the section file matching the TOC entry

If the section audit fails:
- fix the section
- re-audit
- do not advance to the next TOC entry

Receipt placement while Fog remains active:
- If the system is generating Codex artifacts during incubation, receipts belong in the appropriate ambient continuity surfaces and Control Sync outputs.
- This is not the same as governed Quest execution.

--------------------------------------------------------------------------------
6.3 Stitch + verification (Codex Full)
--------------------------------------------------------------------------------

After all section files exist:
- deterministically stitch TOC + sections into <Subject>_Codex_Full.txt

Verification is mandatory before any claim of completion:
- completeness check: every TOC entry has a matching section file
- tail check: the full Codex ends cleanly
- seam check: no duplicates / no gaps

(Use the Large Artifact Composition protocol when applicable.)

--------------------------------------------------------------------------------
6.4 Codex Freeze (Fog Lifted)
--------------------------------------------------------------------------------

Codex Freeze is illegal until:
- the TOC exists
- all required sections for the Codex spine exist
- the full stitch exists
- stitch verification is performed

Freeze procedure:
1) Brains explicitly declares the Codex frozen
2) write: <Subject>_Data/Codex/CODEX_FREEZE.md
   - include date/time
   - include Brains approval

If CODEX_FREEZE.md does not exist, governed execution remains forbidden.

================================================================================
7) CODEX UPDATES (CONTROLLED CHANGE)
================================================================================

Codex changes are allowed, but MUST be deliberate.

Rules:
- Codex changes occur during a Control Sync for the Subject.
- The Control Sync Snapshot MUST record:
  - what changed
  - why it changed
  - what downstream work is affected

If the Codex is already frozen and the change affects structure, constraints, or meaning:
- Brains must explicitly re-authorize
- update CODEX_FREEZE.md accordingly

Routine ambient growth while the Codex is still under Fog does not require pretending the Codex is frozen.
It requires truthful tracking of what is now known and what remains unstable.

================================================================================
8) USING THE CODEX (REFERENCES)
================================================================================

Guild Orders, Dungeons, and Quests MUST reference the Codex by TOC/section numbering.

Rule:
- If work cannot be traced back to a Codex section (or an explicitly deferred decision), it is not execution-ready.

Clarification — anchors are about intent class, not exact nouns:
- Codex anchors do NOT require that the exact feature name appears in the Codex
  (example: “Discord Door” may not be named).
- Anchors MUST point to the governing Codex intent class and constraints
  (example: “Doors / UI interaction channels / Interfaces”).

Vision Delta (classification rule used during Quest creation and Pre-Quest alignment):
- ALIGNED: the work is explicitly described by the Codex sections referenced.
- VARIATION (allowed): the work is a new instance/channel that fits an existing
  Codex intent class (example: add Discord alongside Telegram).
  Required:
  - Declare it as Vision Delta = VARIATION.
  - Obtain Brains authority and codify it via SIDE-QUEST: VISION_VARIATION_CODIFY.
- SHIFT (blocked): the work changes what the Subject IS (identity/purpose).
  Required:
  - Declare it as Vision Delta = SHIFT.
  - Stop and resolve via Control Sync before execution continues.
