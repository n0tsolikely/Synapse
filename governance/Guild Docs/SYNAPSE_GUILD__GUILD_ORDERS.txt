================================================================================
0) PURPOSE OF THIS DOCUMENT
================================================================================

This document defines what **GUILD ORDERS** are, why they exist, and how they are
used.

After reading this document, the phrase:

  “Create Guild Orders.”

MUST be sufficient instruction for any compliant operator (human or AI).

Guild Orders are a core governance primitive.
They define **WHAT** we are committing to build in the current build scope, and
what **DONE** means.

Guild Orders are an artifact.
Conversation is input.
**The Guild Orders file is authority.**

This is governance law.

================================================================================
1) CORE DEFINITION (CANON)
================================================================================

1.1 Guild Orders — definition
Guild Orders are a scope-defining governance artifact that declares the intended
objectives, constraints, and definition of success for a body of work.

Guild Orders answer one question:

> “What are we building in this phase, and how will we know it is done?”

One-sentence definition:
Guild Orders define the authoritative scope and intent of a build effort.

1.2 Relationship to Codex (non-override)
- The Codex defines what the Subject IS (vision, meaning, rules, invariants).
- Guild Orders define what will be BUILT NOW (a bounded subset of Codex intent).

Guild Orders MUST NOT contradict the Codex.
If a conflict is detected:
- stop
- trigger Disclosure Gate
- resolve via Control Sync (Codex clarification or Orders revision)

(See also: `The Guild/Terminology/Terms/Guild_Orders.md`.)

================================================================================
2) WHAT PROBLEM GUILD ORDERS SOLVE
================================================================================

Guild Orders exist because:
- Complex systems cannot be built safely without a shared definition of scope.
- Without declared scope, work drifts into improvisation and contradiction.
- Teams (human or AI) need a stable reference for what “success” means.
- Decisions made in conversation must be elevated into durable intent.

Without Guild Orders:
- work becomes reactive
- priorities shift silently
- “nice ideas” sneak into critical paths
- completion is ambiguous

Guild Orders prevent scope drift by making scope a durable artifact.

================================================================================
3) WHAT GUILD ORDERS ARE — AND ARE NOT
================================================================================

3.1 What Guild Orders ARE
Guild Orders are:
- an intent declaration
- a scope boundary
- a success definition
- a coordination artifact

3.2 What Guild Orders are NOT
Guild Orders are NOT:
- a quest backlog
- a step-by-step plan
- an execution script
- an Execution Pack

Guild Orders do not prescribe HOW work is done.
They declare WHAT must be accomplished and what DONE means.

3.3 Deterministic handoff requirement (anti-ambiguity)
Guild Orders MUST be written so that a separate compliant operator (including a
separate AI session) can:
- derive Dungeons from the Orders (if the Orders were written at a higher level)
- derive atomic Quests from each Dungeon
- execute Quests without inventing intent

Rule:
- If a competent operator must ask “what did you mean?”, then the Orders are
  incomplete and MUST be clarified before execution continues.

================================================================================
4) STRUCTURE: RAIDS AND DUNGEONS (CANON)
================================================================================

4.1 Raids
A **RAID** is the execution state of a Guild Orders document.

Raid states (state by location; see Section 6):
- ACTIVE — Guild Orders are currently being executed
- PAUSED — execution is intentionally halted
- COMPLETED — all objectives are satisfied

A Raid is NOT a section inside the document.
A Raid is the act of executing the Guild Orders.

(See also: `The Guild/Terminology/Terms/Raid.md`.)

4.2 Dungeons
Within Guild Orders, scope is divided into **DUNGEONS**.

A Dungeon is:
- a major slice of scope
- a self-contained objective area
- independently executable if dependencies allow

Dungeons:
- do NOT imply sequence
- do NOT imply priority unless explicitly stated
- may be raided in any dependency-safe order

(See also: `The Guild/Terminology/Terms/Dungeon.md`.)

4.3 Dungeons → Quests (canonical decomposition)
Dungeons decompose into **Quests**.

- A Dungeon may produce one Quest OR many Quests.
- The number of Quests is determined by atomicity.

Atomicity rule:
- If the work cannot be completed and verified independently as a single unit,
  it is NOT a Quest; it is a Dungeon (or larger) and MUST be decomposed.

Quest execution is governed by:
- `Quest Board/SYNAPSE_GUILD__QUEST_VALIDATION_RULES.txt`
- `Processes/SYNAPSE_GUILD__EXECUTION_AUDITS.txt`

================================================================================
4.4 QUEST CREATION VISION ALIGNMENT (NON-OPTIONAL)
================================================================================

When deriving Quests from Dungeons, the Quest Author (human or AI) MUST treat the
Subject as a pre-existing system governed by the Codex (no greenfield assumptions).

Each Quest MUST include the Quest Creation Vision Alignment block defined in:
- `Quest Board/QUEST_TEMPLATE.txt`
- `Quest Board/SYNAPSE_GUILD__QUEST_VALIDATION_RULES.txt`

Minimum requirements at Quest creation:
- Change Class: TRIVIAL / FEATURE / STRUCTURAL (if unsure → STRUCTURAL)
- Vision Delta: ALIGNED / VARIATION / SHIFT
- System Context Statement (what this builds on; platform vs module boundaries)
- Codex Anchors + Constraint Summary (required for execution readiness)

Vision Delta rules:
- ALIGNED: normal execution path.
- VARIATION (allowed): the work is within an existing Codex intent class but is a
  new instance/channel (example: adding a Discord Door alongside Telegram).
  Required:
  - Anchor to the governing Codex intent class.
  - Declare a codification path via SIDE-QUEST: VISION_VARIATION_CODIFY (Hands authority).
- SHIFT (blocked): the work changes what the Subject IS. Execution MUST NOT proceed.
  Resolve via Control Sync (Codex update and/or Orders revision) before continuing.

Repo reality / anti-duplication:
- Pre-Quest Vision Alignment (Orientation + Repo Orientation receipts) is required
  before any state changes, per `Processes/SYNAPSE_GUILD__EXECUTION_AUDITS.txt`.


================================================================================
5) STRUCTURE REQUIREMENTS (MINIMUM SPEC; NO FULL TEMPLATE)
================================================================================

Guild Orders are intentionally not rigidly templated.
The document may be written in whatever shape best communicates intent.

However, Guild Orders MUST still be parseable and unambiguous.

5.1 Minimum required content (Orders-level)
A Guild Orders document MUST include, at minimum:

A) Scope statement
- What is included in this build scope.

B) Definition of Done (Orders-level)
- What it means for the Raid to be COMPLETED.

C) Non-goals / Out of scope
- What is explicitly NOT being built in this scope.

D) Constraints
- Any hard constraints that execution MUST obey (time, quality, architecture,
  compatibility, policy, etc.).

E) Dungeons
- One or more Dungeons defining the major slices of scope.

If any required element is missing or ambiguous:
- the Orders are invalid for execution
- execution MUST NOT proceed
- resolve via Control Sync

5.2 Minimum required content (Dungeon-level)
Each Dungeon MUST include enough detail to derive Quests deterministically.
Minimum required fields per Dungeon:

- Dungeon ID (unique within this Orders document)
- Dungeon Title
- Objective (what changes in the world when this Dungeon is done)
- Definition of Done / Acceptance Criteria (how DONE is proven)
- Dependencies (if any)
- Constraints / Non-goals (Dungeon-specific)
- Artifacts affected (what parts of Engine/Data are expected to change)

If a Dungeon lacks these fields:
- it MUST be clarified before Quests are derived

Note:
- “Artifacts affected” is not a step-by-step plan.
  It is a routing anchor so a Quest Author can find the right surface.

================================================================================
6) RELATIONSHIP TO OTHER ARTIFACTS
================================================================================

6.1 Where Guild Orders live (state by location)
Guild Orders live in Subject Data under:
- `<Subject>_Data/Guild Orders/ACTIVE/`
- `<Subject>_Data/Guild Orders/PAUSED/`
- `<Subject>_Data/Guild Orders/COMPLETED/`

State is defined by location:
- If a Guild Orders file is in ACTIVE, it is eligible for execution.
- If it is in PAUSED, it MUST NOT be executed until resumed.
- If it is in COMPLETED, it is done.

Drafting rule (removes the “missing pre-active state” ambiguity):
- A new Guild Orders document MUST be authored in PAUSED.
- It may move to ACTIVE only when Hands starts the Raid.

Multi-raid rule:
- Multiple raids may exist at once.
- Multiple raids may be ACTIVE at once.
- Control Sync MUST declare which Raid(s) are in scope for the session.

6.2 Completion rule (ACTIVE/PAUSED → COMPLETED)
A Raid is complete when its Dungeons are complete AND verified.
When a Raid completes:
- move the Guild Orders file into `<Subject>_Data/Guild Orders/COMPLETED/`
- record the move in the End-of-Day Snapshot

6.3 Canonical relationships (routing)
- Codex / Canonical Vision:
  Defines what the system is and why it exists.

- Guild Orders:
  Defines what is being built now and what “done” means.

- Quests:
  Atomic commitments created under Dungeons.

- Execution Audits:
  The proof bundle for each Quest.

- Dailies Loop:
  Day-to-day execution rhythm.

- Snapshots:
  Durable continuity of decisions, actions, outcomes, and blockers.

Guild Orders define scope.
They do not replace or override these other artifacts.

================================================================================
7) WHEN TO CREATE GUILD ORDERS (EXECUTION AUTHORITY)
================================================================================

Guild Orders are the canonical scope artifact for build execution.

Rule:
- If Hands intends to execute work across multiple Quests, multiple sessions, or
  multiple operators, Guild Orders MUST exist.

If the intended work is truly trivial and isolated:
- Hands MAY choose to execute it as a single Quest without creating a full
  Guild Orders document, BUT that exception MUST be explicitly declared in
  Control Sync and recorded in the Control Sync Snapshot.

Default:
- If uncertain, create Guild Orders.

================================================================================
8) HOW GUILD ORDERS ARE USED IN PRACTICE (RUNTIME SHAPE)
================================================================================

Canonical lifecycle:

1) Control Sync
- Decisions are made.
- Scope is agreed upon.

2) Guild Orders are authored (PAUSED)
- Scope statement + Done definition are written.
- Dungeons are written using the minimum Dungeon spec.

3) Hands starts the Raid
- Guild Orders file moves PAUSED → ACTIVE.
- Control Sync records the start.

4) Quests are derived
- Atomic work is derived from each Dungeon.
- Quests are written using the Quest Template.

5) Quests are accepted intentionally (Scope Commitment)
- Only Accepted quests may be executed.

6) Execution proceeds via Dailies
- Control Sync → Scope Commitment → Implement (Quest mini-loop) → System
  Verification → Snapshot → Stop/Resume

7) Completion
- All Dungeons are satisfied.
- Guild Orders file moves ACTIVE → COMPLETED.
- Final snapshot records outcomes.

================================================================================
9) ADDING OR MODIFYING SCOPE
================================================================================

Guild Orders may evolve, but changes are governed.

Rules:
- Scope changes require a Control Sync.
- Any scope change MUST be recorded in the Control Sync Snapshot.
- Silent scope changes are forbidden.

Allowed changes:
- New Dungeons may be added if intent expands.
- Existing Dungeons may be clarified or split.
- Definitions of Done may be tightened.

If a change introduces contradiction with the Codex:
- stop
- trigger Disclosure Gate
- resolve before execution continues

================================================================================
10) QUALITY BAR (WHAT MAKES GOOD GUILD ORDERS)
================================================================================

Good Guild Orders:
- make scope explicit
- make success measurable
- prevent random work
- give context to every Quest
- can be handed to a different operator without loss of intent

Bad Guild Orders:
- read like a to-do list
- lack a definition of done
- mix execution steps into intent
- allow “drive-by” additions without review

Handoff test (determinism check):
- If a separate operator can derive the same Dungeons/Quests without asking for
  clarification, the Orders are good.
- If they cannot, the Orders MUST be clarified before execution.

================================================================================
11) CANONICAL COMMANDS
================================================================================

Once this law is accepted, the following phrases are valid:

- “Create Guild Orders.”
Meaning:
Author the authoritative scope and objectives for a build effort.

- “Start a Raid.”
Meaning:
Begin executing the Guild Orders (PAUSED → ACTIVE).

- “Pause the Raid.”
Meaning:
Intentionally halt execution (ACTIVE → PAUSED).

- “Complete the Raid.”
Meaning:
All Dungeons are satisfied; record final snapshot and move to COMPLETED.
