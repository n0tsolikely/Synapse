================================================================================
0) PRIME LAW (TEMPLATES ARE MINIMUMS, NOT LIMITS)
================================================================================
These templates define the MINIMUM structure for Snapshots so they remain readable,
resumable, and audit-safe across days.

Hard rules:
- Snapshot filenames MUST end with the date token (YYYY-MM-DD.txt). No suffix-after-date is allowed.
- Same-day duplicates MUST use a counter BEFORE the date token:
  - <Subject>_<SnapshotType>_Snapshot__02_YYYY-MM-DD.txt (then __03_, etc; zero-padded)
- A Snapshot MUST be an artifact file stored in <Subject>_Data/Snapshots/...
- If the Snapshot file does not exist, then the Snapshot does not exist.
- Templates define minimum required sections.
- The operator MUST append any additional relevant truth even if not listed
  (bugs, failures, blockers, weird behavior, binding decisions, scope changes, etc.).
- These templates MUST NOT be treated as permission to omit reality.

Reference law:
- Guild Docs/SYNAPSE_GUILD__SNAPSHOTS.txt

1) CONTROL SYNC SNAPSHOT — MINIMUM STRUCTURE
================================================================================
Storage location (canon):
- <Subject>_Data/Snapshots/Control Sync/

Filename (canon):
- <Subject>_Control_Sync_Snapshot_YYYY-MM-DD.txt
- If a same-day duplicate is required:
  - <Subject>_Control_Sync_Snapshot__02_YYYY-MM-DD.txt


Mandatory use (canon):
- A Control Sync MUST end with exactly one Snapshot.
  - Control Sync Snapshot if binding decisions exist
  - General Snapshot if no binding decisions exist

Minimum required sections:

A) Header (required)
- Subject:
- Date:
- Topic (optional but recommended):
- Snapshot Type: Control_Sync
- World State (Fog of War / Fog Lifted):
- Codex Freeze status (required when relevant):
  - If Fog Lifted: confirm CODEX_FREEZE marker exists at <Subject>_Data/Codex/CODEX_FREEZE.md
  - If Fog of War: explicitly state “execution forbidden”

B) Rehydration Review (required when resuming work)
Record what was reviewed at the beginning of this Control Sync, in this order:
1) Latest Control Sync Snapshot reviewed (filename)
2) Latest End-of-Day Snapshot reviewed (filename)
3) Relevant General Snapshots reviewed (filenames)
If any expected item is missing, record that explicitly.

C) Current Reality (State Surface)
- Active Guild Orders / Raids:
- Paused Guild Orders / Raids:
- Quest Board state summary:
  - On-board quests:
  - Created Quests or Side-Quests if any:
  - Accepted quests (if any already accepted at start):
  - Blockers:
- Known risks / unknowns:

D) Binding Decisions (required when binding decisions exist)
- Binding decisions made (bullet list)
- Explicitly NOT decided / deferred decisions (bullet list, if relevant)
- Decision changes (if any): “prior decision → new decision” with reference to prior Snapshot/decision

E) Scope Commitment (required)
What the session/day is committing to implement:
- Quests to accept now (by ID) OR statement “no execution committed”
- Any explicitly deferred work (and why)
- Expected side-work (bugs, cleanup, refactors) (if any)

F) Artifacts Touched / Created (required)
List every artifact created/updated during this Control Sync (paths):
- Codex / TOC updates (paths)
- Guild Orders / Raids updates (paths)
- Dungeon / Quest artifacts created/updated (paths)
- Any other <Subject>_Data artifacts touched

G) NOTES(Any other notes or discussion points of interest):

H) Exit Condition + Closure Statement (required)
- Closure statement: “This Control Sync is CLOSED only when this Snapshot exists as a file.”
- If execution will begin: state the first quest to be accepted/executed (or the next immediate action).

2) END-OF-DAY SNAPSHOT — MINIMUM STRUCTURE
================================================================================
Storage location (canon):
- <Subject>_Data/Snapshots/End of Day/

Filename (canon):
- <Subject>_End_of_Day_Snapshot_YYYY-MM-DD.txt
- If a same-day duplicate is required:
  - <Subject>_End_of_Day_Snapshot__02_YYYY-MM-DD.txt


Mandatory trigger (canon):
- When Hands ends a work session (“stop”, “done”, “end session”, or equivalent),
  Brains MUST write an End-of-Day Snapshot.
- If no execution occurred:
  - this Snapshot MUST explicitly state that no execution occurred
  - it MUST reference the latest Control Sync Snapshot (if one occurred)

Minimum required sections:

A) Header (required)
- Subject:
- Date:
- Topic (optional but recommended):
- Snapshot Type: End_of_Day
- World State:
- Codex Freeze status (required when relevant):

B) Work Performed (Quest-by-Quest) (required)
For each Quest worked on (completed OR not), record:
- Quest ID + Title
- Status at end of day (COMPLETED / INCOMPLETE / FAILED / BLOCKED)
- Preconditions referenced (what was true at start; key files/paths reviewed)
- Actions performed (high-level steps; concrete)
- Artifacts changed (paths; engine + data)
- Verification performed (what was run/checked) + receipt locations (audit/log paths)
- Outcome (“what is now true”) OR failure summary

C) Completed Work Summary (required when anything completed)
For each completed Quest:
- Quest ID + Title
- Where proof lives (Execution Audit bundle path or equivalent)
- High-level outcome (“what is now true”)

D) Incomplete / Failed / Blocked Work (required when relevant)
For each incomplete/failed Quest:
- What failed or remained incomplete
- WHY (blocking reason)
- Error output location (Execution Audit / log / receipt path)
- What was attempted (brief)
- Next action required to avoid repeating the failure

E) System Verification Performed (required when applicable)
Record system-level verification beyond per-Quest checks:
- What was run/checked
- Key results
- Any remaining verification gaps

F) Bugs / Weirdness / Discoveries
Record bugs, unexpected behavior, or discovered risks.
If new side-quests were created:
- list the Quest IDs

G) Raid / Guild Orders status changes
- Any raid/order moved ACTIVE → PAUSED (and why)
- Any raid/order moved ACTIVE/PAUSED → COMPLETED (and why)

H) Talent Tree updates (mandatory when relevant)
If any capability was unlocked/expanded today:
- list the Talent Tree update(s)
- reference the Talent Tree log file(s)

I) Resume Instructions (required)
- What tomorrow’s Control Sync must start by reviewing
- Known blockers to address first
- Next recommended Quest(s) to accept OR “requires new Control Sync decision”
- Rehydration note (if applicable): if any rehydration artifacts changed/need update, list paths

Minimum required sections:

A) Header (required)
- Subject:
- Date:
- Topic:

B) Purpose
- What this Snapshot is for

C) Content (facts / notes)
- The content (facts, rationale, observations, design notes, etc.)
- References to Codex/Docs/Quests (paths) if relevant

D) Binding marker (conditional)
- If any statement here is intended to be binding: explicitly mark it “BINDING”.
- Otherwise: do not include binding decisions.

E) NOTES (Any kind of relevant information that was discussed at all, a summery of the majority of the context discussed):

================================================================================
4) NON-NEGOTIABLE QUALITY BAR
================================================================================
Snapshots must be:
- honest (no “we did X” if we didn’t)
- resumable (a competent operator can pick up tomorrow)
- explicit about blockers and failures
- written so that intent does not overwrite reality

Snapshots summarize.
Execution Audits prove.
