## 0) Vocabulary (do not mix concepts)

- **Synapse Guild** = the *body* (operators / team).
- **Synapse** = the *operating system for work* (human + AI), enforced by files + locks + gates.
- **Governance Pack (this folder)** = portable law you inject into an AI session.
- **Subject** = the thing being built (app, book, game, business process, etc.).
- **Hands** = the final authority (the human). When a gate requires Hands, conversation-level “ok” is not enough unless the gate explicitly allows it.

## 1) What this pack does (runtime intent)

When an AI is governed by Synapse, it MUST:

- **Start in Control Sync** (alignment + scope decision) and use the Control Sync Checklist.
- Treat **conversation as input only**. **Filesystem artifacts advance state.**
- Obey **Truth Gate + Disclosure Gate** (no claim without proof; no silent uncertainty).
- Execute build work only through **Dailies** (the per-session execution loop).
- Use **Execution Audits** for every executed Quest (audit bundle: Summary / PreQuest / Execution / Verify / Outcome).
- **Chunk safely** when artifacts are large (Compose → Stitch → Verify → Install → Cleanup).
- Preserve continuity across sessions (Bootstrap Prompt + Continuity Lock + Snapshots).
- **Write snapshots as files** (Control Sync Snapshot on Control Sync close; End-of-Day Snapshot on session end; plus General Snapshots when needed).

If you remember one thing:

> Synapse exists to turn ideas into built reality **without drift, hallucination, or bullshit**.

## 2) The two start paths (choose correctly)

### Path A — Continuing Subject (most common)
You already have a Subject in flight.

**Signal:** you have one or more of:
- `<Subject>_Data/` (Quest Board, Snapshots, Codex, Guild Orders, etc.)
- `<Subject>_Engine/` (code / manuscript / the actual build)

**Gate check (world state):**
- If `<Subject>_Data/Codex/CODEX_FREEZE.md` exists → **Fog Lifted** → execution may be authorized.
- If it does NOT exist → **Fog of War** → treat as Incubation (Path B), even if folders exist.

**Action:** begin at **Control Sync**.

### Path B — New Subject (true beginning / Fog of War)
You have an idea but no frozen Codex.

**Signal:** any of the following are true:
- No `<Subject>_Data/` exists yet
- No Codex exists yet
- `CODEX_FREEZE.md` does not exist

**Action:** run **Subject Initialization + Incubation** under Fog of War until a Codex exists and Hands authorizes Codex Freeze.

**Important:** Under Fog of War, you MAY draft Quests on the Board, but you MUST NOT accept or execute Quests.

## 3) The lifecycle in one page

Authoritative spine:
- `Processes/SYNAPSE_GUILD__THE_LOOP.txt`

### 3.1 Incubation (Fog of War; pre-freeze)
- Create `<Subject>_Data/` skeleton immediately.
- Capture decisions as **Discoveries** (pre-Codex truth units).
- Detect conceptual groupings (features, modes, workflows, constraints, integrations, etc.).
- Run a domain-appropriate **stress test** to eliminate holes.
- Generate a **TOC**.
- Write the **Codex** section-by-section (TOC-first, numbered).
- Stitch + verify the Codex for completeness and internal consistency.
- Perform **Codex Freeze** (Fog Lifted) only when Hands authorizes it (by writing `CODEX_FREEZE.md`).

### 3.2 Build kickoff (Fog Lifted; post-freeze)
- Hands chooses build scope (v1 slice / demo slice / chapter range / etc.).
- Generate **Guild Orders** → break into **Dungeons** → break into **atomic Quests**.
- Populate the **Quest Board**.

### 3.3 Dailies (execution loop; per session/day)
Dailies is the runtime loop for getting work done once a Subject is live.

Every session/day runs:
1) **Control Sync** *(includes deciding scope)*
2) **Scope Commitment** (quest acceptance moves, recorded as artifacts)
3) **Implement** (Quest mini-loop: PreQuest → Execute → Verify → Outcome → state transition)
4) **System Verification** (subject-level verification for the day’s changes)
5) **End-of-Day Snapshot**
6) **Stop / Resume next session**

Scope change rule:
- If scope changes after Control Sync closes, you MUST open a NEW Control Sync and close it with a new Control Sync Snapshot.

## 4) Non-negotiables

### Truth Gate + Disclosure Gate
- No “done” without receipts.
- No claims of executed commands/tests without raw output (no simulated runs).
- If uncertainty affects next steps, you MUST disclose it to Hands.

See: `Processes/SYNAPSE_GUILD__TRUTH_GATE.txt`

- Completion claims require verification evidence.

See: `Processes/SYNAPSE_GUILD__EXECUTION_AUDITS.txt`
