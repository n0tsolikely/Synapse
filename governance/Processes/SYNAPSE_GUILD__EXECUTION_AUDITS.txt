SYNAPSE OS — EXECUTION AUDITS (CANON)
Version: v1.3
Last Updated: 2026-02-09
Status: Governance Law (changes require Governance Control Sync)

Audience:
- Any AI session executing Quests under Synapse OS
- Any human operator (Hands) reviewing or resuming execution
- Any collaborator requiring provable execution truth

================================================================================
0) PURPOSE
================================================================================
Execution Audits exist to preserve provable execution truth.

They ensure work can be:
- understood,
- verified,
- resumed,
- challenged,
- or continued

without relying on memory, assumption, or narrative confidence.

Execution Audits are receipts that bridge:
intention → action → evidence → truth.

They prevent:
- false completion,
- silent failure,
- repeated work,
- unverifiable claims,
- and “we already did this” ambiguity.

They also prevent a specific failure mode:
- hallucinated execution (claiming commands/tests were run without actually running them)

================================================================================
1) SCOPE (WHEN THIS LAW APPLIES)
================================================================================
This law applies whenever:
- A Quest is accepted for execution.
- Files/code/config/system state may be modified.
- Verification is attempted.
- A Quest status changes (Completed / Incomplete / Abandoned).

This law governs execution-phase auditing only.
It does NOT govern:
- Control Sync Snapshots,
- End-of-Day Snapshots,
- Discoveries / Incubation notes,
- Codex content itself.

================================================================================
2) REQUIRED AUDIT PHASES (NON-OPTIONAL)
================================================================================
For every Quest that enters execution, the following phases MUST exist.

No phase may be skipped, assumed, or “rolled into” another phase.

2.1 PRE-QUEST ALIGNMENT AUDIT (MANDATORY)
Trigger:
- Immediately after a Quest is accepted and BEFORE execution begins.

Purpose:
- Prove execution is aligned with Codex/Map, World State, and Locks.

Must include:
- Quest ID + title
- Link(s) to Dungeon/Raid/Guild Order (if applicable)
- Codex sections consulted (explicit references)
- Locks / constraints acknowledged (explicit references)
- Definition of success (what must be true)
- Verification plan (how success will be proven)
- Risks / edge cases / uncertainty
- Confirmation that World State permits execution
  (Codex Freeze marker exists: `<Subject>_Data/Codex/CODEX_FREEZE.md`)

Hard prohibition:
- No file modifications before this exists.

2.2 EXECUTION AUDIT (MANDATORY)
Trigger:
- During execution, as work is performed.

Purpose:
- Record what actually happened (not intent).

Must include:
- Files touched (paths)
- Nature of changes (diffs or explicit description)
- Deviations from plan (if any) + why

Commands/tools invoked (anti-fabrication format):
- For every command/tool you claim you ran, you MUST record:
  - EXACT command line (copy/paste safe)
  - Working directory / context (where it was run)
  - Raw output as receipts (inline if small OR as a separate receipt file path)

Executed vs Proposed Law:
- If a command/tool was NOT actually run, it MUST be labeled PROPOSED (not EXECUTED).
- You MUST NOT present proposed commands as executed work.

Hard prohibition (Truth Gate):
- You MUST NOT invent terminal output, logs, tracebacks, or “PASS” text.

2.3 VERIFICATION AUDIT (MANDATORY)
Trigger:
- After execution attempt and BEFORE changing Quest status.

Purpose:
- Truth-gate the claim that execution succeeded or failed.

Must include:
- Verification steps performed (what checks were actually done)
- Results of each step (PASS/FAIL/BLOCKED per step)
- Determination: overall PASS / FAIL / BLOCKED
- If FAIL: exact failure mode + where + why it blocks completion
- If PASS: what evidence proves it
- If BLOCKED: what could not be executed + why + operator-run command(s) + ledger update (if created)

Rules:
- “Looks correct” is not verification.
- No verification defaults to failure.

2.3.1 VERIFICATION REALITY LAW (NO SIMULATED RUNS)
Definition:
- Verification is valid only if the checks were actually performed.
- Reasoning about expected outcomes is NOT verification.

Rules:
- If you claim a command/test ran, you MUST provide:
  - the exact command
  - the exact raw output (inline or receipt file)
- If the current environment cannot execute verification:
  - you MUST label verification as BLOCKED
  - you MUST trigger Disclosure Gate
  - you MUST run the maximum feasible substitute checks (see VERIFICATION LADDER) and record receipts
  - you MUST provide operator-run command(s) to verify later
  - you MUST update the Blocked Verification Ledger (lazy-created; see VERIFICATION LADDER)
  - you MUST NOT claim a PASS outcome

Failure output law:
- If verification fails, you MUST preserve failure evidence:
  - full traceback / failing assertion output (or receipt file path)
  - names of failing tests/checks (if applicable)

2.3.2 DOMAIN-SPECIFIC VERIFICATION (CODE vs NON-CODE)
Verification MUST be appropriate to the Subject type.

A) CODE / SOFTWARE SUBJECTS
- Follow the Verification Ladder: `Processes/SYNAPSE_GUILD__VERIFICATION_LADDER.txt`
- Prefer the project's real test runner when it exists (pytest, etc.).
- Per-quest verification MUST match the chosen TL and run the smallest correct set of executed checks relevant to the change.
- If there is an existing full test suite and it is feasible to run:
  - you MUST run it at least once per session during “System Verification” (DAILIES) and store receipts
  - per-quest verification may use a targeted subset (as defined in the Verification Plan)

B) NON-CODE / WRITING / DESIGN SUBJECTS
Verification is a deterministic checklist against Codex invariants.
- You MUST cite the Codex sections used as invariants.
- Each invariant MUST be marked PASS/FAIL with concrete evidence:
  - quote(s) of the changed text
  - cross-reference(s) to where it is stored
- A vague statement like “tone is consistent” is invalid.
  You must specify the invariant and how it was checked.

2.3.3 PATCH DISCIPLINE AND DRIFT DETECTION

During Quest execution, Brains MAY apply minor corrective patches without spawning a new Quest ONLY IF all of the following are true:

- The change is small and local (e.g. typo, import fix, single-line logic correction).
- The change is directly required to complete the current Quest.
- The change does NOT alter architecture, scope, or Codex invariants.
- The change does NOT introduce new modules, folders, schemas, or abstractions.
- All required verification checks are re-executed and PASS after the patch.

Such patches MUST be recorded in the Quest Evidence.

If ANY of the following occur, a DRIFT EVENT is detected and Brains MUST immediately halt execution and open an EMERGENCY CONTROL SYNC:

- The fix requires refactoring or non-local changes.
- Multiple fixes cascade from an initial fix (“fixing fixes”).
- The Quest specification is discovered to be incorrect or incomplete.
- New prerequisite work is required outside the Quest’s scope.
- Continuing execution would violate Codex laws or architectural invariants.

During the Emergency Control Sync, Brains MUST explain the drift and propose Side Quests or plan corrections BEFORE any further execution.

No execution may continue beyond a Drift Event without auditable authorization.

2.4 POST-QUEST OUTCOME NOTE (MANDATORY)
Trigger:
- When the Quest leaves execution state.

Purpose:
- Make the outcome resumable without guesswork.

Must include:
- Final status: COMPLETED / INCOMPLETE / ABANDONED
- State transition performed (Accepted → Completed/Incomplete/Abandoned), including paths moved
- Audit bundle path (canonical location)
- If INCOMPLETE: blocker + what was attempted + why continuation failed
- If ABANDONED: justification + follow-up required (yes/no)
- Recommended next actions

================================================================================
3) STORAGE LAW (WHERE EXECUTION AUDITS LIVE)
================================================================================
Execution Audits live in the Subject Data folder (never in Engine, never in Governance):

<Subject>_Data/Audits/Execution/

Canonical structure: one audit bundle folder per Quest.

Example:
<Subject>_Data/Audits/Execution/
  QUEST_014__2026-02-02__<short_slug>/
    00_SUMMARY.md
    01_PREQUEST.md
    02_EXECUTION.md
    03_VERIFY.md
    04_OUTCOME.md
    10_DIFF.patch                  (optional)
    11_COMMAND_OUTPUT__*.txt       (optional)
    12_TEST_OUTPUT__*.txt          (optional)
    20_ARTIFACT_LOCATIONS.txt      (optional)

Minimum required bundle contents:
- 00_SUMMARY.md   (MANDATORY, human-readable; the audit must not be an unreadable wall of receipts)
- 01_PREQUEST.md  (MANDATORY)
- 02_EXECUTION.md (MANDATORY)
- 03_VERIFY.md    (MANDATORY)
- 04_OUTCOME.md   (MANDATORY)

Notes:
- Additional receipts are optional but recommended when useful (diffs, logs, outputs).
- If an output is too large for a single audit file, write it as a separate receipt file and reference it.

Legacy compatibility:
- Older Subjects may use date-folder trees like: <Subject>_Data/Audits/Execution/YYYY/YYYY-MM/
- This is allowed as legacy, but the audit bundle structure above is canonical going forward.

================================================================================
3.1) MULTI-DAY QUESTS & RETRIES (SAME BUNDLE, APPEND-ONLY LAW)
================================================================================
If a Quest spans multiple Dailies (or requires retries):
- The Quest remains in <Subject>_Data/Quest Board/Accepted/ until it is truly verified and complete.
- The audit bundle folder path is canonical. DO NOT create a second bundle for the same Quest.
- Instead, append a new time-stamped attempt block inside the SAME audit phase files.

Append format (deterministic):
- Add a heading like:
  ## YYYY-MM-DD — Attempt N
- Under it, record:
  - what changed since the last attempt
  - what was re-verified
  - the new result (PASS/FAIL)

Outcome Note rule:
- It may start as INCOMPLETE.
- When the Quest is later completed, append a final “Completion Proof” block:
  - reference the PASS verification attempt
  - reference the artifact(s) produced

Hard prohibition:
- A Quest may not move to Completed unless the latest executable verification outcome is either:
  - PASS, OR
  - BLOCKED with Disclosure Gate triggered, operator-run command(s) recorded, and Blocked Verification Ledger updated.

================================================================================
4) RELATIONSHIP TO SNAPSHOTS
================================================================================
Execution Audits are granular proof.
Snapshots are continuity summaries.

- Control Sync Snapshot = decisions/intent
- End-of-Day Snapshot = outcomes/blockers

Snapshots MUST reference Execution Audits when claiming execution.
Snapshots do not replace audits.

================================================================================
5) RELATIONSHIP TO TRUTH + DISCLOSURE
================================================================================
Truth Gate enforces: “No claim without proof.”
Execution Audits are the primary proof artifacts.

If required audits are missing:
- Quest state may not advance
- Completion is forbidden
- Disclosure Gate must notify Hands
- Execution pauses until compliance is restored
