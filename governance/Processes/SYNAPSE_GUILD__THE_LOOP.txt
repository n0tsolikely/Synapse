# SYNAPSE OS — THE LOOP
*(Idea → Codex → Execution → Completion → Repeat)*
Version: v1.3
Last Updated: 2026-02-06
Status: Governance Law (changes require Governance Control Sync)


This document defines Synapse as a deterministic **pipeline with nested loops**.

- **Outer pipeline loop (project lifecycle):**
  Idea → Incubation → TOC → Codex → Codex Freeze → Build (Guild Orders → Dungeons → Quests) → Completion/Iterate

- **Inner execution loops (build-time):**
  - **Dailies Loop (per session/day):** Control Sync → Scope Commitment → Implement (Quest Mini-Loop) → System Verification (when applicable) → Snapshot → Stop/Resume
  - **Quest Mini-Loop (per accepted quest):** Pre-Quest Alignment → Execute → Verify → Outcome → Filesystem State Transition

If any gate is unmet, execution is **ILLEGAL** and the system MUST stop and disclose.

---

## 0. NON-NEGOTIABLE INVARIANTS (ALWAYS TRUE)

### 0.1 Authority Order (anti-drift)
- Conversation is **input**.
- Files, locks, and state artifacts are **authority**.
- If conversation conflicts with state, state wins.

### 0.2 World State Gate (execution legality)
- **Fog of War:** no execution. Only incubation / TOC / Codex work is allowed.
- **Fog Lifted:** execution is permitted only after Codex Freeze is valid.

Machine-check:
- Fog Lifted requires: `<Subject>_Data/Codex/CODEX_FREEZE.md`

### 0.3 Truth + Disclosure Gates (claim legality)
- No claims without proof (Truth Gate).
- If proof cannot be produced, Disclosure Gate MUST trigger and the action is BLOCKED.

### 0.4 Canonical Boundaries (no parallel state)
- One canonical working tree.
- One canonical `<Subject>_Data/`.
- One canonical `<Subject>_Engine/`.

If multiple competing copies exist, the system MUST HALT until Hands selects canon.

## 1. INCUBATION (NO CODEX; FOG OF WAR)

The Loop always begins with **an idea**.

At this point:
- there is no Codex
- there may be no Engine
- nothing is executable
- everything is exploratory

This is the **Incubation Phase**.

### 1.1 Entering Incubation
The system enters Incubation when:
- Hands wants to start something new, OR
- Hands is describing an idea and no Codex exists for the subject.

Incubation is the default state when a Codex does not exist.

### 1.2 Subject Creation (immediate)
As soon as a subject exists, the system MUST:
- identify the subject (name + key)
- create the subject’s `<Subject>_Data/` skeleton (canonical)

Purpose:
- preserve continuity
- prevent loss across chat boundaries

### 1.3 Incubation Behavior (build forbidden)
During Incubation, the system MUST:
- prompt for clarification and expansion
- capture requirements, constraints, risks, and decisions as they emerge

During Incubation, the system MUST NOT:
- modify `<Subject>_Engine/` (execution)
- claim implementation progress

### 1.4 Continuous Capture (no loss rule)
As ideas and decisions emerge, the system MUST write them into subject-state artifacts.

These incubation artifacts:
- are allowed to change
- are allowed to contradict earlier artifacts
- are not enforceable law

Their job is memory and traceability, not correctness.

### 1.5 Structural Awareness (make TOC possible without guessing)
As incubation continues, the system MUST notice structure forming and cluster information accordingly.

The structure is domain-agnostic. Examples:
- software: workflows, permissions, integrations
- books: characters, world rules, arcs, chapters
- games: mechanics, progression loops, systems
- essays: claims, evidence, counterarguments

Rule:
- the system MUST avoid parallel conflicting versions of the same concept.
- if conflicts exist, they MUST be called out explicitly and resolved by Hands.

### 1.6 Periodic Crystallization (continuity anchors)
To prevent long-session blur, the system MUST periodically crystallize:
- what is decided
- what is open
- what changed since last crystallization

Crystallizations are:
- not Codex
- not enforceable
- continuity anchors only

Crystallize at minimum:
- before a session ends
- after major decision changes
- when explicitly requested

### 1.7 Stress Testing (optional; Fog of War safe)
At any point during Incubation, Hands MAY request a stress test.

Definition:
- a stress test is a structured thought experiment that runs the subject end-to-end (or long-horizon) to discover missing laws, edge cases, contradictions, or unasked questions.

Rules:
- stress testing is allowed only in Fog of War (Incubation). It does not authorize execution.
- any discovered gaps MUST be captured into incubation artifacts.
- if critical gaps are found, Incubation continues.

## 2. FROM IDEA TO MAP — TABLE OF CONTENTS (TOC)

Incubation ends only when Hands explicitly directs TOC/Codex creation.

Hands intent examples:
- “Make the TOC”
- “Write the Codex”
- “Freeze the map”

### 2.1 Mandatory Gate: Final Completeness Stress Test (pre-TOC)
Before creating the TOC, the system MUST run a final completeness stress test.

Purpose:
- discover critical holes before the map becomes authoritative

Stress test mode depends on subject type:
- software/programs: long-horizon use-case thought experiment
- books/stories/games: end-to-end narrative/system simulation
- any subject: list remaining gaps explicitly

If critical gaps are found:
- the system MUST capture them
- Incubation continues

Only when there are no critical gaps may the system proceed to TOC creation.

### 2.2 Creating the TOC (the map)
The system derives the TOC from:
- incubation artifacts
- crystallizations
- the emerged structure

Rules:
- the TOC represents the system **as if it already exists**
- the TOC MUST be complete enough to write the Codex without guessing
- truncation is forbidden (chunking rules apply)

Output:
- a TOC artifact that becomes the authoritative map for Codex writing

---

## 3. CODEX WRITING (MAP → LAW)

With the TOC in place, the system writes the Codex.

Rules:
- Codex is written section-by-section and subsection-by-subsection.
- Codex text MUST be unambiguous.
- Codex MUST be consistent with the TOC numbering and titles.

### 3.1 Self-Audit Gate (before every section)
Before writing a section, the system MUST:
- re-read the TOC
- re-read relevant incubation material
- confirm scope and numbering

### 3.2 Section Writing + Audit Gate (after every section)
After writing a section, the system MUST audit for:
- alignment with the TOC
- omission (missing required detail)
- hallucinated detail
- truncation / accidental summarization

If truncation or summarization is detected:
- the section MUST be split
- written in multiple parts
- stitched
- re-audited

### 3.3 Codex Continuity Across Sessions (no drift)
If a session ends mid-Codex:
- the system MUST update the subject’s continuity artifacts (Latest Rehydration Pack)
- it MUST record:
  - where the system left off
  - what sections are complete
  - what comes next

### 3.4 Codex Completion Gate (stitch + review)
Codex Freeze must not occur on an incomplete Codex.

When all TOC sections are written, the system MUST:
- stitch the Codex into a single canonical artifact (or a single canonical stitched view)
- audit the stitched Codex for completeness and internal consistency
- resolve any discovered gaps before freezing

Only after completion audit passes may Hands freeze the Codex.

---

## 4. CODEX FREEZE (EXECUTION AUTHORIZED)

Codex Freeze is the execution gate.
It transitions World State from Fog of War to Fog Lifted.

Codex Freeze is valid only when:
- Hands explicitly declares the Codex frozen, AND
- the marker file exists: `<Subject>_Data/Codex/CODEX_FREEZE.md`

Without the marker:
- World State remains Fog of War
- execution is illegal

---

## 5. BUILD KICKOFF (CONTROL SYNC → GUILD ORDERS)

Before build execution begins, the system MUST run a Control Sync.

Control Sync outcomes (minimum):
- rehydrate (review snapshots + current state)
- confirm World State (Fog of War vs Fog Lifted)
- decide build intent and scope

Scope choices:
- full build, OR
- a versioned subset (e.g., “v1” / “one chapter” / “a vertical slice”)

That scope becomes the **Guild Orders**.

Decomposition topology (must be respected):
- Guild Orders contain **Dungeons** (major phases)
- Dungeons decompose into **Quests** (atomic executable work)
- Quests are placed onto the Quest Board and may be accepted for execution

If `<Subject>_Engine/` does not exist yet:
- create it as a stub
- all execution work occurs in the Engine from this point forward

---

## 6. EXECUTION (DAILIES; QUEST MINI-LOOP)

Dailies are the execution-phase loop used when:
- a subject exists
- `<Subject>_Data/` is in play
- `<Subject>_Engine/` exists (or is created at kickoff)
- World State permits execution (Fog Lifted)

### 6.1 Dailies Topology (per session/day)
A Daily MUST follow this structure:

1) Control Sync (alignment + rehydration)
2) Scope Commitment (which quests are accepted for this session)
3) Implement (Quest Mini-Loop for each accepted quest)
4) System Verification (when applicable)
5) Snapshot (end-of-day continuity)
6) Stop (until next session)
7) Resume (next session begins again at Control Sync)

The canonical daily execution protocol is defined in:
- `Processes/SYNAPSE_GUILD__DAILIES.txt`

### 6.2 Quest Mini-Loop (per accepted quest)
For each accepted quest, the system MUST execute the audit-driven mini-loop:
- Pre-Quest Alignment Audit
- Execute smallest correct changes
- Execution Audit
- Verification + Outcome
- Filesystem state transition (Board move)

This is governed by:
- `Processes/SYNAPSE_GUILD__EXECUTION_AUDITS.txt`
- `Quest Board/SYNAPSE_GUILD__QUEST_VALIDATION_RULES.txt`

Rule:
- Only quests in `Accepted/` may be executed.

If World State is Fog of War:
- Dailies may run only as conceptual planning.
- No Engine modifications.
- No quest execution.

---

## 7. COMPLETION AND ITERATION

When all quests for the current Guild Orders are complete:
- return to Control Sync
- decide next scope (new Guild Orders), OR declare completion

State evidence:
- move the Guild Orders/Raid document from ACTIVE → COMPLETED
- record this transition in the End-of-Day snapshot

This continues until the subject is finished.

When the subject is complete:
- a final snapshot is written
- completion is declared with evidence (not narration)

---

## 8. WHY THIS WORKS

This loop enforces:
- no execution without understanding
- no understanding without capture
- no progress without proof
- no drift across sessions

The AI is not a narrator.
It is an operator.

End.
