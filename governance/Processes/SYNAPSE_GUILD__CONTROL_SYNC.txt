================================================================================
0) PURPOSE OF THIS DOCUMENT
================================================================================

This document defines what a CONTROL SYNC is, why it exists, and how it is used.

After reading this document, the phrase “Control Sync” MUST be sufficient instruction
for any compliant operator (human or AI) to enter Control Sync mode.

A Control Sync is the primary decision‑making + alignment ritual of the Synapse Guild.

This is governance law.

================================================================================
1) CORE DEFINITION (CANON)
================================================================================

1.1 Control Sync — definition
A Control Sync is a structured meeting held by the Synapse Guild to align thinking,
surface reality, make decisions, and deliberately evolve work, scope, or governance.

A Control Sync answers one core question:

> “What decisions are we making now, and what becomes true because of them?”

One‑sentence definition:
A Control Sync is the authoritative forum where Synapse Guild decisions are made,
validated, or reaffirmed.

1.2 Control Sync is a MODE (stateful), not a message
A Control Sync is considered ACTIVE until Hands explicitly ends it.

- Switching into writing/editing mode does NOT end a Control Sync.
- Creating or editing artifacts may occur inside an active Control Sync.

Hard rule:
Brains MUST NOT assume the Control Sync has ended due to silence, topic drift,
or the start of drafting.

================================================================================
2) WHY CONTROL SYNCS EXIST
================================================================================

Control Syncs exist because:

- Complex work requires alignment before execution.
- Decisions made casually tend to drift or be forgotten.
- Teams (human or AI) need a shared moment to reset context.
- Important decisions must have a traceable origin.

Without Control Syncs:
- work becomes reactive
- scope shifts invisibly
- decisions are re‑litigated
- execution loses coherence

Control Syncs keep the system sane.

================================================================================
3) WHAT A CONTROL SYNC IS — AND IS NOT
================================================================================

3.1 What a Control Sync IS
A Control Sync is:
- a meeting
- a thinking space
- a decision forum
- a context reset
- an alignment checkpoint

3.2 What a Control Sync is NOT
A Control Sync is NOT:
- a task list
- a rigid agenda
- a daily stand‑up script
- an execution session
- a snapshot itself

A Control Sync may produce artifacts, but it is not the artifacts.

--------------------------------------------------------------------------------
3.3 CONTROL SYNC ACTIVATION (TRIGGERS)
--------------------------------------------------------------------------------

A Control Sync MUST be considered engaged when Hands uses any clear phrasing
indicating a sync is starting, including but not limited to:
- "Control Sync"
- "Hold a Control Sync"
- "Control Sync start"
- "Control Sync engage"
- "Begin Control Sync"
- equivalent natural-language phrasing

If the phrasing is ambiguous (e.g., quoting the term), Brains MUST ask:
"Is that a Control Sync start?" and WAIT.

--------------------------------------------------------------------------------
3.4 CONTROL SYNC START SEQUENCE (AI/HUMAN)
--------------------------------------------------------------------------------

Upon activation, Brains MUST perform the following in order.
This is the minimum deterministic entry protocol.

STEP A — Identify Subject + Mode (Continuing vs New)
- If the current Subject is already known and <Subject>_Data exists:
  - declare the Subject name
  - declare canonical paths (Subject_Data + Subject_Engine if applicable)
- If the Subject is NOT known OR Subject_Data cannot be found:
  - Brains MUST stop and ask Hands the minimum entry questions:
    1) What is the Subject name/key?
    2) Is this a NEW Subject or a CONTINUING Subject?
    3) If continuing: where is <Subject>_Data located?
    4) If there is an engine/codebase: where is <Subject>_Engine located?

Hard rule:
- If the Subject cannot be identified, Brains MUST NOT proceed into execution.
- If Hands indicates "New Subject": route to
  Processes/SYNAPSE_GUILD__SUBJECT_INITIALIZATION_AND_INCUBATION.txt

STEP B — Capture Control Sync metadata
Brains MUST capture and state:
- date
- start time
- participants
- Subject
- reason for sync (rehydration / planning / decisions / governance)

STEP C — State the current continuity anchor
Brains MUST state what the current authoritative anchor is, using what exists:
- Latest Continuity Lock / Bootstrap Prompt (if subject-state exists)
- Latest Snapshots (Control Sync → End-of-Day → relevant General)

If artifacts are missing:
- state the missing artifacts explicitly
- do NOT invent them

STEP D — Rehydrate current state (minimum)
Brains MUST summarize:
- where we left off
- unfinished work
- known issues/bugs/blockers
- key ongoing discussions/decisions
- why things may not have been finished

If Guild Orders/Raids are referenced during rehydration:
- list ONLY TITLES
- label Active vs Paused
- do NOT auto-generate, rewrite, or decompose

STEP E — Run deterministic minimum checks (when execution is possible)
If the Control Sync may lead into execution (Dailies / Quests / engine edits),
Brains MUST run:
- Processes/SYNAPSE_GUILD__CONTROL_SYNC_CHECKLIST.txt

This is the ignition key that prevents:
- missing-state execution
- Fog of War violations
- unvalidated quest acceptance

STEP F — STOP + WAIT (Hands authority)
After completing the start sequence, Brains MUST STOP and WAIT.

Brains may offer a short set of next-step options, but MUST NOT advance without Hands.

Strict prohibitions unless Hands explicitly commands otherwise:
- DO NOT auto-generate new Guild Orders
- DO NOT auto-start a Raid
- DO NOT auto-break into Quests
- DO NOT auto-advance into Dailies execution

--------------------------------------------------------------------------------
3.5 CONTROL SYNC CLOSEOUT (ENDING)
--------------------------------------------------------------------------------

A Control Sync remains active until Hands explicitly ends it.

Valid end phrases include (non-exhaustive):
- "End Control Sync"
- "Close Control Sync"
- "Control Sync done"
- "We are done with Control Sync"

On explicit end, Brains MUST:
1) Confirm scope/decisions that are now binding (or explicitly deferred).
2) Ensure a Snapshot is written when required by Snapshot law + checklist.
   - If binding decisions were made: Control Sync Snapshot
   - If alignment/context only: General Snapshot (recommended)
   - Dailies ignition (session start) requires a Control Sync Snapshot.
3) Restate what the next system phase is (e.g., proceed to Scope Selection / Dailies).

Hard rule:
Brains MUST NOT treat Control Sync as ended until the closeout is complete.

================================================================================
4) TYPES OF CONTROL SYNCS
================================================================================

Control Syncs are intentionally flexible.
They may vary in depth, formality, and output.

Common types include:

4.1 Alignment / Catch‑Up Control Sync
- Rehydrate context after time away.
- Review current state.
- Surface what has changed or been forgotten.
- No major decisions required.

Often captured as a General Snapshot.

4.2 Exploratory / Ideation Control Sync
- Discuss new ideas or directions.
- Explore possibilities without commitment.
- No obligation to produce Quests or Guild Orders.

Outputs may be notes or a General Snapshot.

4.3 Decision‑Making Control Sync
- Make binding decisions.
- Generate or update:
  - Codex documents
  - Guild Orders
  - Execution Packs
  - Quests
- Lock scope and intent.

Typically produces a Control Sync Snapshot.

4.4 Governance Control Sync
- Modify the operating system itself.
- Update canon documents.
- Introduce or retire rituals.

Must produce a Snapshot (Control Sync Snapshot if binding; General if alignment-only).

================================================================================
5) WHAT HAPPENS INSIDE A CONTROL SYNC
================================================================================

A Control Sync may include any of the following activities:

- Reviewing current reality (state, progress, blockers)
- Re‑reading key artifacts (Guild Orders, Quests, Talent Tree)
- Surfacing problems or confusion
- Discussing ideas or alternatives
- Making decisions (or explicitly deferring them)
- Creating or refining:
  - Guild Orders
  - Quests
  - Execution Packs
  - Canon documents
- Deciding daily or near‑term scope

There is no required conversational script.
The constraint is intentionality, not structure.

================================================================================
6) DECISIONS AND AUTHORITY
================================================================================

All binding decisions must originate from a Control Sync.

Rules:
- If a decision matters later, it must be made or confirmed in a Control Sync.
- If a decision is made, it must be recorded in a snapshot.
- If a decision is deferred, that deferral must be explicit.

Control Syncs are where authority is exercised deliberately, not implicitly.

================================================================================
7) CONTROL SYNCS AND ARTIFACT GENERATION
================================================================================

Control Syncs are the birthplace of many artifacts, including:

- Guild Orders (scope commitments)
- Quests (atomic execution commitments)
- Execution Packs (anti‑drift execution law)
- Canon documents (governance law)
- Continuity Locks (state sealing for chat boundaries)

Not every Control Sync produces every artifact.
But every artifact that matters should trace back to a Control Sync.

================================================================================
8) CONTROL SYNCS AND SNAPSHOTS
================================================================================

A Control Sync itself is ephemeral.
Its consequences are not.

After a Control Sync:
- Important outcomes must be captured in a Snapshot.

Snapshot types commonly associated:
- Control Sync Snapshot — when binding decisions were made.
- General Snapshot — when the sync captured context without binding decisions.

The snapshot is the record.
The Control Sync is the moment.

================================================================================
9) CONTROL SYNCS AND DAILIES
================================================================================

Control Syncs are part of the Dailies loop, but they are not limited to daily use.

In Dailies:
- Control Sync aligns the day/session.
- Scope may be chosen.
- Execution follows.

Outside Dailies:
- Control Syncs may happen anytime alignment or decisions are needed.

Control Syncs are not constrained by calendar.
They are constrained by necessity.

================================================================================
10) QUALITY BAR (WHAT MAKES A GOOD CONTROL SYNC)
================================================================================

A good Control Sync:
- reduces confusion
- clarifies reality
- makes decisions explicit
- leaves participants aligned

A bad Control Sync:
- avoids decisions without naming deferral
- produces vague outcomes
- leaves scope ambiguous
- is not recorded when it should be

The goal is clarity, not ceremony.

================================================================================
11) CANONICAL COMMANDS
================================================================================

Once this law is accepted, the following phrases are valid:

- “Control Sync.”
Meaning:
Enter Control Sync mode immediately.

- “Hold a Control Sync.”
Meaning:
Convene a deliberate meeting to align, decide, or rehydrate context.

- “This requires a Control Sync.”
Meaning:
The issue is important enough to warrant explicit discussion and decision.

- “Record this as a Control Sync Snapshot.”
Meaning:
The outcomes of the meeting must be captured as binding record.

- “End Control Sync.”
Meaning:
Run closeout (including required snapshot), then permit progression into the next phase.
