# SYNAPSE OS — SUBJECT INITIALIZATION + PRE-CODEX INCUBATION
*(Beginning State / Idea Formation / Non-Executable)*

---

## 0) PRIME PURPOSE

Synapse OS must support a true beginning:
- user has an idea
- there is no subject workspace yet
- there is no Codex yet
- and nothing must be lost

This document defines:
1) how to initialize a Subject workspace from nothing
2) how incubation behaves until a Codex exists
3) how the system produces a Codex deterministically (no guessing)

---

## 1) SUBJECT INITIALIZATION (MANDATORY WHEN SUBJECT DOES NOT EXIST)

### 1.1 Subject Identification
When the user starts a new idea, the system must establish:
- Subject name (human readable)
- Subject key/slug (filesystem-safe identifier)

Example:
- Subject name: "Stuart"
- Subject key: "Stuart"

The Subject key is used for folder naming:
- `<Subject>_Data`
- `<Subject>_Engine` (created at build kickoff, unless already provided)

### 1.2 Create Canonical Subject Root
If it does not already exist, the system MUST create:

- `<Subject>_Data/`  (stateful continuity + governance artifacts + idea capture)

The system MUST NOT create `<Subject>_Engine/` during Incubation **unless** one of the following is true:
- Hands provides an existing engine/repo and requests it be treated as canonical, OR
- Hands explicitly requests an Engine be created before Codex Freeze (rare).

Default rule:
- New Subject starts with **Data only**.
- Engine is created at **Build Kickoff (post-Codex Freeze)**.

Important:
- “Engine” does NOT mean “code only.”
- Engine means “the thing being built.”
  Examples:
  - software → code repo
  - book → manuscript source
  - game → project files
  - essay → the essay itself

### 1.3 Create Mandatory `<Subject>_Data` Skeleton

Lock reference:
- The `<Subject>_Data` minimum skeleton is locked.
- See: The Guild/Terminology/Locks/LOCK__SUBJECT_DATA_SKELETON.md
These directories MUST exist for every subject, regardless of domain:

#### A) Snapshots
- `<Subject>_Data/Snapshots/General/`
- `<Subject>_Data/Snapshots/Control Sync/`
- `<Subject>_Data/Snapshots/End of Day/`

#### B) Guild Orders
- `<Subject>_Data/Guild Orders/ACTIVE/`
- `<Subject>_Data/Guild Orders/PAUSED/`
- `<Subject>_Data/Guild Orders/COMPLETED/`

#### C) Quest Board (Filesystem State Machine)
- `<Subject>_Data/Quest Board/`
- `<Subject>_Data/Quest Board/Accepted/`
- `<Subject>_Data/Quest Board/Completed/`
- `<Subject>_Data/Quest Board/Abandoned/`

Rule:
- Quests “on the board” may live directly in:
  `<Subject>_Data/Quest Board/`
- Once accepted, they must move into `Accepted/`.
- Completed goes to `Completed/`.
- Abandoned requires Control Sync decision.

(Quest templates may exist in Quest Board root as needed.)

#### D) Codex
- `<Subject>_Data/Codex/`
- `<Subject>_Data/Codex/Sections/`

Codex artifacts:
- `<Subject>_Data/Codex/TOC.md` (or TOC_DRAFT.md during incubation)
- `<Subject>_Data/Codex/CODEX_FREEZE.md` (exists only after Hands declares Codex Freeze)
- `<Subject>_Data/Codex/Sections/<numbered_sections>.md` (one subsection per file)
- `<Subject>_Data/Codex/<Subject>_Codex_Full.txt` (TOC + all sections stitched into one canonical full-text deliverable)

Rule:
- Sections are written individually (chunk-safe).
- FULL is generated by deterministic stitch after verification.

#### E) Docs
- `<Subject>_Data/Docs/`
- `<Subject>_Data/Docs/Notes/`

Purpose:
- Notes are non-canon brainstorming and supporting explanation.
- Notes may be messy. They exist to prevent loss of thought.
- Notes do not replace Codex once Codex exists.

#### F) Buffs (Ignition System)
- `<Subject>_Data/Buffs/`

The subject MUST contain the canonical Buff set:
1) `<SUBJECT>_EXECUTION_PROTOCOL.txt`
2) `<SUBJECT>_DATA_DIRECTORY_MAP.txt`
3) `<SUBJECT>_SESSION_START_CHECK.txt`

Rule:
- Buffs are canonical session startup documents.
- They are read at session start to prevent drift.
- Copies may appear in rehydration packs, but canonical Buffs live here.

#### G) Talent Tree
- `<Subject>_Data/Talent Tree/`

#### H) To Do
- `<Subject>_Data/To Do/`
(Non-binding idea list. Not quests.)

#### I) Latest Rehydration Pack
- `<Subject>_Data/Latest Rehydration Pack/`

Purpose:
- This is a convenience export bundle for starting a fresh chat or resuming fast.
- It may contain:
  - Continuity Lock
  - Bootstrap Prompt
  - Latest Snapshots
  - Copies of Buffs
  - Any other session resume material
- It is reference-only unless Hands explicitly orders edits inside it.

#### J) Incubation
- `<Subject>_Data/Incubation/`

Incubation contains the live “idea capture” artifacts:
- `<Subject>_Data/Incubation/INCUBATION_LEDGER.md`
- `<Subject>_Data/Incubation/INCUBATION_CRYSTALS.md`
- `<Subject>_Data/Incubation/CLUSTERS/` (optional; for grouped notes as structure emerges)

Rule:
- Incubation artifacts are not canon.
- They exist to make Codex formation deterministic.

#### K) Execution Audits (Subject-local, canonical)
- `<Subject>_Data/Audits/Execution/`
(Execution audits live in subject data, not the engine.)

## 2) PRE-CODEX INCUBATION PHASE (NO CODEX EXISTS)

### 2.1 Purpose
Incubation is the idea formation stage where:
- exploration is allowed
- structure is allowed to emerge
- execution is forbidden

The goal is to fully understand the subject end-to-end so that a Codex can be written
“as if the thing already exists.”

### 2.2 Entry Conditions
Incubation is entered when:
- no Codex exists for the subject, and
- the user is exploring or describing an idea.

### 2.3 Execution Prohibition (Non-Negotiable)
While in incubation:
- do not generate Guild Orders
- do not generate Quests
- do not implement or execute
- do not build the Engine deliverable

(Subject bootstrap is allowed. It is scaffolding, not project execution.)

### 2.4 Continuous Capture Obligation (No Loss Rule)
During incubation, the system MUST continuously capture:
- decisions
- constraints
- requirements
- assumptions
- risks
- dependencies
- any meaningful design direction

Capture must be detailed enough that the idea can be reconstructed later
without relying on chat memory.

Capture location is deterministic:
- `<Subject>_Data/Incubation/INCUBATION_LEDGER.md`

### 2.5 Structural Coherence Obligation (Domain-Agnostic)
The system MUST actively detect and maintain emerging structure, regardless of project type.

This means:
- identify recurring conceptual clusters (whatever they are for the subject)
- keep related ideas grouped over time
- when decisions change, update the same cluster rather than forking parallel conflicting ideas
- avoid “scattered drift” that makes TOC creation guessy

The system must not force a fixed taxonomy.
Structure must emerge from the subject itself.

### 2.6 Periodic Crystallization (Continuity Anchors)
The system MUST periodically write “crystals” that summarize:
- what changed since the last crystal
- current understanding
- open questions
- emerging clusters/structure

Crystals are written to:
- `<Subject>_Data/Incubation/INCUBATION_CRYSTALS.md`

Trigger examples:
- after extended discussion
- after major decision changes
- before ending a session
- when explicitly requested by Hands

### 2.7 Control Sync Snapshots During Incubation
During incubation, Control Sync snapshots are especially important.

They must:
- be detailed
- reference incubation artifacts
- describe how the idea is evolving as understanding increases (“fog lifting”)
- record the current structural shape (emerging clusters)

Snapshots must be saved to:
- `<Subject>_Data/Snapshots/Control Sync/`

## 3) EXIT CONDITIONS — CODEX FORMATION

Incubation ends when Hands explicitly says they are ready to formalize, such as:
- “Generate the TOC”
- “Write the Codex”
- “Freeze the map”
- “We’re ready to build”

At this point, the system:
1) performs a **final completeness sweep** (domain-appropriate stress test)
   - goal: identify missing decisions, unresolved questions, or holes
   - outcome: either fill the gaps, or explicitly record them as deferred
2) derives a TOC from incubation structure
3) writes Codex sections one-by-one (section/subsection)
4) self-audits each written section against the TOC for numbering + scope + completeness
5) stitches all sections + TOC into `<Subject>_Codex_Full.txt`
6) runs a final stitch verification (tail + seam sanity)
7) if Hands approves, writes the **Codex Freeze marker**:
   - `<Subject>_Data/Codex/CODEX_FREEZE.md`
   - includes date/time + Hands approval

Execution remains forbidden until Codex Freeze is declared **and the marker exists**.

---

## 4) POST-CODEX EXECUTION (OUT OF SCOPE HERE)

Once Codex exists and is frozen (Fog Lifted):
- If `<Subject>_Engine/` does not exist yet, create it at **Build Kickoff**.
- Then proceed: Codex → Guild Orders → Quests → Dailies → Audits → Snapshots

This is the standard execution pipeline already defined elsewhere.

## 5) SUBJECT-SPECIFIC EXTENSIONS (OPTIONAL, DOMAIN-DEPENDENT)

In addition to the mandatory skeleton, subjects MAY add domain folders under `<Subject>_Data/` when useful.

Examples:
- book project:
  - `<Subject>_Data/Characters/`
  - `<Subject>_Data/World/`
  - `<Subject>_Data/Factions/`
- business system:
  - `<Subject>_Data/Integrations/`
  - `<Subject>_Data/Policies/`
- game:
  - `<Subject>_Data/Mechanics/`
  - `<Subject>_Data/Assets/`
  - `<Subject>_Data/Progression/`

Rule:
- these are support artifacts for Codex creation and engine writing.
- they do not replace the mandatory governance skeleton.
