# SYNAPSE — SUBJECT INITIALIZATION + INCUBATION (FOG OF WAR)
*(Beginning State / Existing Repo Adoption / Pre-Freeze Structure Formation)*

---

## 0) PRIME PURPOSE

Synapse must support a true beginning in both of the ways a real Subject begins:
- Brains has an idea and there is no Subject workspace yet, OR
- Brains already has an Engine/repo and wants Synapse to attach without pretending the project began today.

At initialization time:
- there may be no `<Subject>_Data` yet
- there may be no frozen Codex yet (Fog of War)
- the Subject may still be poorly defined
- work may already exist in an Engine
- nothing gets lost
- governed execution does not begin early

This document defines:
1) how to initialize a Subject workspace from nothing or by attaching to an existing Engine
2) how incubation behaves while World State is Fog of War (pre-freeze)
3) how the system transitions from captured ideas and emerging structure → TOC → Codex → Codex Freeze deterministically

Startup clarification:
- `<SUBJECT>` is documentation notation only.
- Literal `Subject` / `<SUBJECT>` must never be used as a real active runtime subject.
- Active subject selection comes from:
  - `python3 runtime/synapse.py engage`
  - `python3 runtime/synapse.py focus`
  - `python3 runtime/synapse.py resolve-subject`
  - `python3 runtime/synapse.py attach-or-init`
- If a prior focus lock exists, startup should continue, switch, or create a subject rather than silently defaulting to placeholder values.

## 1) SUBJECT INITIALIZATION (MANDATORY WHEN <Subject>_Data DOES NOT EXIST)

### 1.1 Subject identification (binding)

A Subject MUST be identified by two values:
- Subject name (human readable)
- Subject key (slug; filesystem-safe namespace)

Hard rule:
- The Subject key is a binding namespace choice. It MUST be explicitly confirmed by Brains.

Slug constraints (minimum; deterministic):
- ASCII
- NO spaces
- Allowed characters: A–Z a–z 0–9 _ -
- Examples:
  - ✅ <SUBJECT>
  - ✅ subject_os
  - ✅ VideoGame-V1
  - ❌ “My Project” (contains spaces; must be rewritten)
  - ❌ “weird/thing” (contains slash; must be rewritten)

The Subject key is used for folder naming:
- `<Subject>_Data`
- `<Subject>_Engine` (created at build kickoff unless one already exists and Brains adopts it as canonical)

### 1.2 Initialization paths (greenfield vs adoption)

Subject initialization now has two lawful entry paths:

**A) Greenfield initialization**
- there is no existing canonical Engine yet
- Synapse creates `<Subject>_Data/`
- Codex/Legend/structure begin from capture and drafting
- any early motion remains under Fog of War

**B) Existing Engine / repo adoption**
- Brains provides an existing Engine/repo and requests it be treated as canonical
- Synapse creates `<Subject>_Data/` beside or alongside that Engine
- the existing Engine becomes the canonical Engine root for the Subject
- Synapse MUST NOT pretend the project began at adoption time

Adoption rule:
- Existing work may be inspected, mapped, and summarized
- inferred understanding from code/history/docs is PROVISIONAL until Brains confirms or corrects it
- adoption must not fabricate false past Quests, false past Guild Orders, or false historical certainty

Strong recommendation (not a block):
- After adoption, Brains SHOULD hold an alignment / Control Sync to confirm:
  - what the Subject is
  - current purpose and scope
  - major capabilities already present
  - what is still unclear or provisional

### 1.3 Create canonical Subject root (Data-first rule)

If it does not already exist, Hands MUST create:
- `<Subject>_Data/`  (stateful continuity + governance artifacts + idea capture)

Default rule (non-negotiable unless Brains overrides):
- New Subjects start with **Data only**.
- `<Subject>_Engine/` is created at **Build Kickoff (post-Codex Freeze)**.

Hands MUST NOT create `<Subject>_Engine/` during Fog of War unless one of the following is true:
- Brains provides an existing engine/repo and requests it be treated as canonical, OR
- Brains explicitly requests an Engine be created before Codex Freeze.

Clarification:
- “Engine” does NOT mean “code only.”
- Engine means “the thing being built.”
  Examples:
  - software → code repo
  - book → manuscript source
  - game → project files
  - essay → the essay itself

### 1.4 Existing Engine / repo adoption (deterministic)

When adopting an existing Engine/repo, Hands MUST:
1) preserve the Engine as the canonical deliverable root
2) create the canonical `<Subject>_Data/` continuity root
3) create the mandatory Subject skeleton + Buffs + startup continuity artifacts
4) establish the ambient sidecar surfaces under `<Subject>_Data/.synapse/`
5) treat all inferred understanding as provisional until corrected or stabilized

Hands MUST NOT, during adoption:
- invent false historical Quests as if they were witnessed in real time
- invent false historical Guild Orders / Raids from weak evidence
- claim Codex Freeze from existing code alone
- claim governed execution readiness merely because an Engine already exists

Hands MAY, during adoption:
- derive a provisional Codex seed
- derive provisional subsystem/capability summaries
- derive quest candidates / side-quest candidates from current work and deterministic runtime signals
- prepare a truthful adoption snapshot or Control Sync anchor

### 1.5 Create mandatory `<Subject>_Data` skeleton (locked)

Lock reference (authoritative):
- The Guild/Terminology/Locks/LOCK__SUBJECT_DATA_SKELETON.md

Rule:
- The skeleton MUST exist exactly as required by the lock.
- Additional subject-specific folders are allowed, but the locked minimum skeleton MUST remain present.

Minimum required structure (restated for convenience; lock wins on conflict):

```
<Subject>_Data/
  Snapshots/
    General/
    Control Sync/
    End of Day/

  Guild Orders/
    ACTIVE/
    PAUSED/
    COMPLETED/

  Quest Board/
    Accepted/
    Completed/
    Abandoned/
    # Quest files may also live at the root of Quest Board when "on the board"

  Audits/
    Execution/

  Codex/
    TOC.md
    TOC_DRAFT.md            # optional during incubation
    CODEX_FREEZE.md         # exists only after Brains freezes the Codex
    Sections/
    <Subject>_Codex_Full.txt

  Docs/
  Buffs/
  To Do/
  Talent Tree/
  Latest Rehydration Pack/
  Incubation/
  Archive/
```

Additional notes (allowed):
- `<Subject>_Data/Docs/Notes/` is permitted for non-canon notes.
- `<Subject>_Data/Incubation/CLUSTERS/` is permitted for grouped incubation notes.

### 1.6 Required Buffs (Ignition set)

Inside `<Subject>_Data/Buffs/`, the canonical Buff set MUST exist:
1) `<SUBJECT>_EXECUTION_PROTOCOL.txt`
2) `<SUBJECT>_DATA_DIRECTORY_MAP.txt`
3) `<SUBJECT>_SESSION_START_CHECK.txt`

Rule:
- Buffs are canonical session startup docs to prevent drift.
- Copies may appear in rehydration packs, but canonical Buffs live here.

Additional requirement (code/software subjects):
- `<SUBJECT>_SESSION_START_CHECK.txt` MUST include the verification command slots defined in:
  - `Continuity/SYNAPSE_GUILD__BUFFS.txt`
- At minimum, include:
  - `VERIFY_SMOKE_CMD: ...`
  - `VERIFY_FULL_CMD: ...`
  - `VERIFY_E2E_CMD: ...` (optional)
  - `NETWORK_POLICY: ...`
  - `EXECUTION_SURFACE: ...`

Early-stage allowance:
- If tests do not exist yet, these fields MUST still exist and be set to `NONE YET`.
- In that case, the Session Start Check MUST specify the baseline executable verification to run
  (e.g., TL1 static/import checks) until real tests exist.
- This is a truthful placeholder that MUST be updated once tests/doors exist.

## 2) INCUBATION (FOG OF WAR; PRE-FREEZE)

### 2.1 Definition

Incubation is the Fog of War phase where:
- exploration is allowed
- structure is allowed to emerge
- Codex drafting is allowed
- Build Manual drafting is allowed
- ambient quest and side-quest candidate formation is allowed
- governed execution is forbidden

Goal:
- define the Subject deeply enough that the Codex spine can become stable
- preserve all meaningful motion while the Subject is still clarifying itself
- avoid false certainty while still allowing the system to learn what the Subject is

### 2.2 Entry conditions (deterministic)

World State is Fog of War unless Codex Freeze has occurred.

Gate condition for Fog Lifted (both must be true):
1) Brains explicitly declares the Codex frozen, AND
2) the marker file exists:
   - `<Subject>_Data/Codex/CODEX_FREEZE.md`

Therefore, Incubation / Fog of War applies when:
- `<Subject>_Data` exists but `<Subject>_Data/Codex/CODEX_FREEZE.md` does NOT exist, OR
- `<Subject>_Data` does not exist yet (new Subject; initialization + incubation are required).

Hard rule:
- A drafted TOC does not lift Fog.
- A drafted Codex does not lift Fog.
- A complete-looking codebase does not lift Fog.
- Only the freeze marker lifts Fog.

### 2.3 Governed execution prohibition (non-negotiable)

While World State is Fog of War:
- DO NOT accept Quests
- DO NOT start a Raid
- DO NOT move Guild Orders into ACTIVE as governed work
- DO NOT claim `governed_execution_ready`
- DO NOT present exploratory movement as already-authorized governed history

Clarification:
- Fog of War blocks **governed execution**.
- Fog of War does NOT require inactivity.

### 2.4 Allowed motion during incubation

Allowed during Fog of War:
- create the Subject_Data skeleton
- attach Synapse to an existing Engine/repo
- write/update Incubation artifacts
- draft TOC and Codex sections
- draft Build Manual material
- draft Guild Orders in `PAUSED`
- draft Quests and Side-Quests on `BOARD` when truthfully supported
- run ambient capture / sidecar updates
- record decisions, discoveries, disclosures, run state, changed files, and verification surfaces
- perform exploratory or prototype implementation/prototyping if it remains explicitly non-governed

Hard rule:
- exploratory implementation under Fog MUST NOT be mistaken for accepted quest execution
- exploratory implementation under Fog MUST drag its continuity story alongside it

That means if code or other Engine work occurs during Fog, Synapse MUST also preserve:
- what moved
- why it moved
- what it appears to be building toward
- what remains provisional or unresolved

If movement occurs but the continuity story is not kept, Disclosure Gate MUST trigger.

### 2.5 Continuous capture obligation (No Loss Rule)

During incubation, Hands MUST continuously capture:
- decisions
- constraints
- requirements
- assumptions
- risks
- dependencies
- any meaningful design direction
- any exploratory implementation that changes understanding of the Subject

Capture must be detailed enough that the idea can be reconstructed later without relying on chat memory.

Canonical capture surfaces during incubation may include:
- `<Subject>_Data/Incubation/INCUBATION_LEDGER.md`
- `<Subject>_Data/Incubation/INCUBATION_CRYSTALS.md`
- ambient sidecar ledgers under `<Subject>_Data/.synapse/`
  - decisions
  - discoveries
  - run state
  - proposal/candidate state

Rule:
- long-form human-readable incubation capture and runtime sidecar capture may coexist
- they MUST NOT silently diverge

### 2.6 Structural coherence obligation (domain-agnostic)

Hands MUST actively detect and maintain emerging structure.

This means:
- identify recurring conceptual clusters (whatever they are for the subject)
- keep related ideas grouped over time
- when decisions change, update the same cluster rather than forking conflicting parallel truths
- avoid “scattered drift” that makes TOC creation guessy
- if deterministic runtime signals reveal a bounded work unit, allow that work unit to become a quest candidate rather than leaving it unstructured

The system MUST NOT force a fixed taxonomy.
Structure must emerge from the subject.

### 2.7 Periodic crystallization (continuity anchors)

Crystals are condensed checkpoints that summarize:
- what changed since the last crystal
- current understanding
- open questions
- emerging structure / clusters
- whether exploratory motion changed the Subject meaningfully

Crystals live at:
- `<Subject>_Data/Incubation/INCUBATION_CRYSTALS.md`

Minimum required cadence (deterministic):
- update crystals at the end of any Control Sync that occurs during Fog of War (before Control Sync closeout)
- update crystals immediately before generating the TOC
- update crystals before ending a session if meaningful incubation occurred without a Control Sync
- update crystals when exploratory implementation materially changes Subject understanding and no other canonical capture surface has summarized it yet

If Hands cannot write/update crystals:
- Disclosure Gate MUST trigger
- do NOT proceed as if capture succeeded

### 2.8 Control Sync snapshots during incubation

Control Sync still governs decisions during Fog of War.

Rule:
- Snapshot requirements are governed by Snapshot law.
- During incubation, Control Sync Snapshots MUST also record:
  - what incubation artifacts were updated (paths)
  - what ambient sidecar truth changed materially
  - current cluster/structure summary
  - what remains ambiguous / deferred
  - whether any exploratory implementation occurred under Fog and what it changed

Snapshot location (canonical):
- `<Subject>_Data/Snapshots/Control Sync/`

## 3) EXIT CONDITIONS — TOC + CODEX FORMATION → CODEX FREEZE

Incubation ends only when Brains explicitly commands formalization, such as:
- “Generate the TOC”
- “Write the Codex”
- “Freeze the map”
- “We’re ready to build”

When this occurs, Hands MUST follow Codex law:
- `Guild Docs/SYNAPSE_GUILD__CODEX.txt`

Minimum deterministic sequence (high-level):
1) Completeness sweep
   - identify missing decisions / unresolved questions / holes
   - either fill them OR record them explicitly as DEFERRED (do not hand-wave)
2) Derive TOC from captured structure (ledger + crystals + current canonical sidecar truth; not guesses)
3) Write Codex sections one-by-one (one TOC entry per section file)
4) Audit each section against the TOC (numbering + scope + completeness; no placeholders)
5) Stitch TOC + sections into `<Subject>_Codex_Full.txt`
6) Verify the stitch (no gaps, no duplicates, clean end)
7) If Brains approves: write the Codex Freeze marker:
   - `<Subject>_Data/Codex/CODEX_FREEZE.md`
   - include date/time + Brains approval

Hard rule:
- Governed execution remains forbidden until Codex Freeze is declared AND the marker exists.

Clarification:
- Freeze does not require that all future detail has already been written.
- Freeze requires that the Codex spine is stable enough for governed execution.

## 4) POST-FREEZE EXECUTION (OUT OF SCOPE HERE)

Once Codex exists and is frozen (Fog Lifted):
- If `<Subject>_Engine/` does not exist yet, create it at Build Kickoff.
- Then proceed:
  Codex → Guild Orders → Dungeons → Quests → Dailies → Audits → Snapshots

This is defined elsewhere (The Loop + Dailies + Guild Orders + Quest law).

In addition to the mandatory skeleton, Subjects MAY add domain folders under `<Subject>_Data/` when useful.

Examples:

  - `<Subject>_Data/Characters/`
  - `<Subject>_Data/World/`
  - `<Subject>_Data/Factions/`
- business system:
  - `<Subject>_Data/Integrations/`
  - `<Subject>_Data/Policies/`
- game:
  - `<Subject>_Data/Mechanics/`
  - `<Subject>_Data/Assets/`
  - `<Subject>_Data/Progression/`

Rules:
- subject-specific folders MUST NOT replace or delete the locked skeleton.
- if a new folder becomes a recurring anchor, it SHOULD be referenced/defined in the Codex.
