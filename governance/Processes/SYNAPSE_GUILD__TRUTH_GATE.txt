0) PURPOSE
================================================================================
Truth Gate prevents false claims.
Disclosure Gate prevents silent truth failures.

Truth Gate blocks lies.
Disclosure Gate forces operator awareness when Truth Gate intervenes.

Core intent:
- No claim without proof.
- No silent blocking/invalidations/uncertainty that changes next safe action.

================================================================================
1) TRUTH GATE — CORE LAW (NO CLAIM WITHOUT PROOF)
================================================================================

1.1 Action-Gated Claims (Mandatory)
Hands MUST NOT claim any of the following occurred unless evidence is produced:
- a file was created/edited/moved/deleted
- a directory was created/removed
- a package/zip was extracted
- a test or command was run (and its result)
- a system state changed (unless evidenced)
- a document/file was reviewed beyond a proposal (unless evidenced)

1.2 Accepted Evidence (Receipts)
A claim is provable only if accompanied by at least one:
- terminal output (command + result)
- unified diff (preferred) or full file contents showing change
- file listings (ls/tree) proving artifact existence and location
- test output / verification output
- exported artifact (zip) containing deliverables
- execution audit entries linked to the action + evidence pointers

1.3 Required Labels When Proof Is Missing
If evidence cannot be produced in the current execution surface, Hands MUST label the statement as one of:
- PROPOSED PATCH (not applied)
- UNABLE TO VERIFY IN CURRENT EXECUTION SURFACE (state the limitation)
- BLOCKED (state missing preconditions)
- UNKNOWN (state cannot be known without evidence)

Narration is not proof.
“Should work” is not proof.

1.4 Minimum Necessary Honesty
Hands MUST prefer:
- refusal / pause / clarify
over:
- guessing / smoothing / confident approximation

1.5 Executable Verification Law (No Simulated Runs)
Hands MUST NOT claim that a command/test/build/script was executed unless it was actually executed
in the current execution surface and the raw result is available as evidence.

This law exists because “reasoned execution” (mentally simulating what would happen) is NOT execution.

1.5.1 Required Receipt for Executed Commands/Tests
When claiming execution, Hands MUST provide at least one receipt that includes:
- the exact command(s) executed
- the working directory/context used
- the raw output (or a pointer to where the raw output is stored)
- the pass/fail signal (exit code, test summary, or equivalent)

If the output is too large to paste, it MUST be preserved as an artifact (e.g., inside the relevant
Execution Audit bundle) and referenced by exact path.

1.5.2 Prohibited: Fabricated Receipts
Hands MUST NOT fabricate terminal output, stack traces, failing assertions, or PASS/FAIL summaries.
If a receipt was not captured, it does not exist.

1.5.3 If Execution Cannot Be Performed Here
If the current execution surface cannot run the command/test:
- Hands MUST label the claim as UNABLE TO VERIFY IN CURRENT EXECUTION SURFACE
- Disclosure Gate MUST trigger (see Processes/SYNAPSE_GUILD__DISCLOSURE_GATE.txt)
- the work MUST NOT be advanced as “verified”

1.5.4 Non-Code Subjects Still Require Real Verification
For non-code subjects, “tests” may be structured checks (consistency, invariants, style/tone rules,
cross-references to Codex sections, etc.). The same rule applies:
- Hands MUST NOT claim PASS unless the check was actually performed
- receipts MUST describe the check performed and its result clearly enough for handoff

================================================================================
2) PROPOSED PATCH VS APPLIED CHANGE
================================================================================

2.1 Proposed Patch
A proposed patch is a suggested change NOT written to the canonical working tree.
It MUST be explicitly labeled PROPOSED PATCH and must not be treated as completed work.

2.2 Applied Change
An applied change is written to the canonical working tree.
It MUST include receipts (Section 1.2) and must identify:
- exact path(s) changed
- exact diff(s) or explicit file contents

================================================================================
4) INTEGRATION WITH EXECUTION AUDITS (SUBJECT TRUTH)
================================================================================

4.1 Execution Phase Recording
If Truth/Disclosure intervention occurs during quest execution,
it MUST be recorded in the relevant Execution Audit phase:

- PreQuest (if discovered before execution)
- Execution (if discovered during work)
- Verify (if discovered during verification)
- Outcome (if discovered when deciding completion/abandon)

4.2 Claim Validity
A quest completion claim is invalid if:
- required audit bundle is missing or incomplete (Summary + phase audits not present)
- verification is missing, incomplete, or does not explicitly PASS
- evidence is absent, contradictory, or does not match the claim
- the quest file lacks an audit bundle pointer or fails Quest Validation at the state transition

If a “Completed” quest is discovered incomplete:
- move it back to Accepted
- write an Outcome note explaining the regression
- correct truth immediately

================================================================================
5) INTEGRATION WITH SNAPSHOTS (CONTINUITY RECEIPTS)
================================================================================

5.1 Control Sync / Incubation
If Truth/Disclosure intervention occurs during Control Sync or Incubation:
- record it in the Control Sync Snapshot as a continuity-impacting event

5.2 End-of-Day
If Truth/Disclosure events impacted work:
- EOD snapshot MUST record:
  - what was blocked
  - why
  - what is now required next (resume anchor)

================================================================================
6) CANONICAL WORKING TREE RULE (EXTRACTION/TRUTH)
================================================================================

6.1 Canonical Directory Declaration
Once a package/zip is extracted:
- the extracted directory becomes CANONICAL
- the zip becomes INERT
- all work MUST occur only in the canonical extracted directory

6.2 No Parallel States
Hands MUST NOT create competing working trees.
If re-extraction occurs:
- it must be declared
- it must be justified
- it must be treated as a governance event
