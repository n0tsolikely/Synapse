# SYNAPSE GUILD — DAILIES (v1.3)

Last Updated: 2026-02-02

================================================================================
0) PURPOSE
================================================================================

Dailies are the repeating execution loop used to turn Quests into completed work **without drift**.

Dailies exist so that the system reliably:
- starts with alignment (Control Sync)
- executes only what was accepted
- produces receipts (audits + diffs + tests)
- updates quest state deterministically
- updates capability state (Talent Tree when applicable)
- produces end-of-day continuity

================================================================================
1) SCOPE + WORLD STATE
================================================================================

Dailies are an **execution-phase** loop.

- If World State is **Fog of War** (no Codex Freeze marker), Dailies may run only as **conceptual planning**.
  - No engine modifications.
  - No Quest execution.

- If World State is **Fog Lifted** (Codex Freeze marker exists), Dailies are authorized.

Machine-check:
- Fog Lifted requires: `<Subject>_Data/Codex/CODEX_FREEZE.md`

================================================================================
2) ENTRY CONDITION
================================================================================

Dailies begin only after:
1) a Control Sync has occurred
2) Hands has accepted a scope (explicit or implied by accepted quests)
3) World State permits execution (Fog Lifted + Codex Freeze marker)

================================================================================
3) THE 7 DAILY STEPS (CANON)
================================================================================

A Daily execution loop MUST follow these steps in order:

--------------------------------------------------------------------------------
3.1 STEP 1 — CONTROL SYNC (ALIGNMENT)
--------------------------------------------------------------------------------

Mandatory:
- Run the Control Sync Checklist:
  - `Processes/SYNAPSE_GUILD__CONTROL_SYNC_CHECKLIST.txt`

Minimum outcomes:
- Rehydrate by reviewing snapshots in locked order:
  1) latest Control Sync Snapshot (intent/decisions)
  2) latest End-of-Day Snapshot (execution reality/outcomes/blockers)
  3) relevant General Snapshots (if any)
- Review current Quest Board state.
- Review Active/Paused Guild Orders/Raids.
- Confirm World State (Fog of War vs Fog Lifted).
- Decide the day’s intended scope.

Control Sync is governed by:
- `Processes/SYNAPSE_GUILD__CONTROL_SYNC.txt`

Output:
- A Control Sync Snapshot when the Control Sync ends.

--------------------------------------------------------------------------------
3.2 STEP 2 — SCOPE SELECTION (COMMITMENT)
--------------------------------------------------------------------------------

If execution is permitted (Fog Lifted):
- Select which quests will be executed today/session.

Acceptance Validity Gate (mandatory):
For each quest being accepted:
1) Validate the Quest file format
   - `Quest Board/SYNAPSE_GUILD__QUEST_VALIDATION_RULES.txt`
2) Create the Quest’s audit bundle folder under:
   - `<Subject>_Data/Audits/Execution/`
3) Create the minimum required audit phase files inside the bundle (stubs allowed)
4) Record the audit bundle folder path inside the Quest file
5) Move the Quest file into:
   - `<Subject>_Data/Quest Board/Accepted/`

Rules:
- Only quests in `Accepted/` may be executed.
- A quest that cannot be executed today should remain on the board (or be explicitly paused/abandoned by Hands).

--------------------------------------------------------------------------------
3.3 STEP 3 — IMPLEMENTATION (PER-QUEST MINI-LOOP)
--------------------------------------------------------------------------------

For each accepted quest in scope, the system MUST perform the following mini-loop:

A) Pre-Quest Alignment Audit (MANDATORY)
- Create the audit before changing any files.
- Store it inside the quest’s audit bundle:
  - 01_PREQUEST.md
- Confirm:
  - quest goal
  - relevant Codex sections
  - relevant locks/constraints
  - success criteria
  - World State permits execution

B) Execute Changes
- Make the smallest correct set of changes.
- Record files touched and commands run.

C) Execution Audit (MANDATORY)
- Record what actually happened inside the audit bundle:
  - 02_EXECUTION.md
- Include:
  - file paths
  - diffs or explicit change detail (or link to patch file)
  - commands and outputs
  - deviations

--------------------------------------------------------------------------------
3.4 STEP 4 — VERIFICATION + OUTCOME (PER-QUEST MINI-LOOP)
--------------------------------------------------------------------------------

For each quest, after implementation:

A) Verification Audit (MANDATORY)
- Run appropriate verification (tests, lint, smoke run, compilation, etc.).
- Record the steps and results in:
  - 03_VERIFY.md
- If outputs are large, store as separate receipt files and reference them.

B) Post-Quest Outcome Note (MANDATORY)
- Record final status in:
  - 04_OUTCOME.md
- If incomplete: exact blocker + next action.
- Update 00_SUMMARY.md with a human-readable summary.

C) State Transition (FILESYSTEM STATE MACHINE)
- If completed (latest verification attempt is PASS):
  - move quest file to `<Subject>_Data/Quest Board/Completed/`
- If abandoned:
  - move quest file to `<Subject>_Data/Quest Board/Abandoned/`
- If incomplete:
  - keep in `Accepted/` (or pause explicitly under Hands direction)

Hard rule:
- A quest may NOT be moved to Completed without required audits AND an explicit PASS verification result.

--------------------------------------------------------------------------------
3.5 STEP 5 — TALENT UPDATE (WHEN APPLICABLE)
--------------------------------------------------------------------------------

If a quest created a new durable capability:
- update the subject’s Talent Tree (and/or Buffs if instructed)

Rule:
- Talents record capabilities; they are not proof.
- Proof lives in audits + tests.

--------------------------------------------------------------------------------
3.6 STEP 6 — SNAPSHOT + CONTINUITY (END OF DAY)
--------------------------------------------------------------------------------

At end of the Daily session:

A) End-of-Day Snapshot (mandatory)
- Write an End-of-Day Snapshot that captures:
  - what happened today (gist)
  - what succeeded
  - what failed / remained incomplete (and WHY)
  - bugs/weirdness encountered
  - verification performed
  - talent updates
  - raid status changes
- Snapshots are governed by:
  - `Guild Docs/SYNAPSE_GUILD__SNAPSHOTS.txt`
  - `Guild Docs/SYNAPSE_GUILD__SNAPSHOT_TEMPLATES.txt`
  (Templates are minimums, not limits.)

B) Raid lifecycle updates (when applicable)
- If a Raid/Guild Order is fully complete:
  - move it into `<Subject>_Data/Guild Orders/COMPLETED/`
- If a Raid is intentionally paused:
  - move it into `<Subject>_Data/Guild Orders/PAUSED/`

C) Latest Rehydration Pack update (when applicable)
- If this work will cross chat boundaries or days:
  - update `<Subject>_Data/Latest Rehydration Pack/`
  - include continuity lock + bootstrap prompt as needed
  - include references to latest snapshots + quest state surfaces
- Follow the Reference-Only rule (do not duplicate the entire project unless required).

--------------------------------------------------------------------------------
3.7 STEP 7 — STOP / REPEAT
--------------------------------------------------------------------------------

- Stop when Hands ends the day.
- Resume tomorrow with a new Control Sync.

================================================================================
4) NON-NEGOTIABLE RULES
================================================================================

- No execution without Codex Freeze marker.
- No quest acceptance without Quest Validation + audit bundle pointer.
- No file changes without Pre-Quest Alignment Audit.
- No quest completion claims without Verification Audit + explicit PASS.
- If Truth Gate blocks, Disclosure Gate triggers immediately.
- Templates are minimums; capture any additional important detail needed for truth + continuity.
