SYNAPSE OS — SCHEMA VALIDATION (CANON)
Version: v1.0
Last Updated: 2026-02-12
Status: Governance Law (changes require Governance Control Sync)

Audience:
- Any AI session using SYNAPSE_STATE.yaml for routing/rehydration
- Any runtime implementation of Synapse OS
- Hands reviewing whether routing/state manifests are safe to rely on

================================================================================
0) PURPOSE
================================================================================
Synapse OS uses YAML manifests as deterministic routing + state pointers.

If these manifests drift silently (fields renamed, structures changed, partial
writes), the runtime can:
- mis-route required files,
- select the wrong “latest” artifact,
- or treat ambiguous state as valid.

Schema Validation exists to prevent that class of failure.

================================================================================
1) CORE LAW
================================================================================
1.1 State manifests MUST have a formal schema contract
The following manifests MUST have an explicit, versioned schema contract:
- Governance: `SYNAPSE_STATE.yaml`
- Subject: `<Subject>_Data/SUBJECT_STATE.yaml` (subject-local pointer registry)

Canonical schema contracts (governance-level):
- `Schemas/SYNAPSE_STATE.schema.json`
- `Schemas/SUBJECT_STATE.schema.json`

1.2 Validation is required BEFORE execution reliance
If execution, routing, or rehydration depends on these manifests, validation MUST
occur before claiming deterministic correctness.

Minimum requirement:
- `SYNAPSE_STATE.yaml` MUST validate against `Schemas/SYNAPSE_STATE.schema.json`.
- If `<Subject>_Data/SUBJECT_STATE.yaml` exists, it MUST validate against
  `Schemas/SUBJECT_STATE.schema.json`.

1.3 Proof requirement (anti-handwaving)
Brains MUST NOT claim that routing/state manifests are valid unless a validation
receipt exists for the current manifest contents.

================================================================================
2) WHEN VALIDATION IS REQUIRED
================================================================================
Validation MUST be performed:
- whenever `schema_version` changes in either manifest
- whenever routing rules / required_read_order changes
- whenever a Subject’s root pointers change (data_root / engine_root / pointers)
- before the first Quest execution in any session that will rely on routing

================================================================================
3) RECEIPTS (EVIDENCE)
================================================================================
A validation receipt MUST be captured.

Allowed receipt locations (choose one, but be explicit):
A) Paste raw validator output into the current Control Sync Snapshot
   under a section titled: `Schema Validation`.
B) Store a receipt file in Subject Data, referenced from the current session
   snapshot and/or the current Quest audit bundle.

Receipt MUST include:
- which manifest(s) were validated
- which schema file(s) were used
- PASS/FAIL outcome
- raw validator output (or a path to the raw output)

================================================================================
4) FAILURE BEHAVIOR
================================================================================
If validation FAILS:
- Disclosure Gate MUST trigger
- execution authority is INVALID
- no Quest execution is permitted
- only analysis / documentation / Codex work may continue until repaired

If validation cannot be performed in the current execution surface:
- Disclosure Gate MUST trigger
- execution authority is INVALID
- the session MUST switch to PROPOSED PATCH / OPERATOR REQUIRED behavior
  (see Truth Gate + Mode Awareness)

================================================================================
5) IMPLEMENTATION NOTE (NON-BINDING)
================================================================================
This law is tool-agnostic.

Any validator that can parse YAML → JSON and validate JSON Schema (Draft 2020-12)
may be used, provided receipts are captured.

END OF DOCUMENT
